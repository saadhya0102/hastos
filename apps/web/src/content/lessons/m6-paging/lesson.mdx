# Pages, Page Tables, and the TLB

Virtual memory is implemented at the granularity of fixed-size blocks called **pages**. The OS and
hardware maintain **page tables** that map virtual pages to physical frames, and a cache called the
**TLB** keeps translation fast. This is the machinery behind every memory access your program makes.

## Pages and frames

Memory is divided into fixed-size **pages** (virtual) and **frames** (physical), typically **4 KiB**
each. A virtual address splits into two parts:

```
| virtual page number (VPN) | page offset |
```

For 4 KiB pages, the **offset is the low 12 bits** (2^12 = 4096); the rest is the page number.
Translation maps the VPN → a physical frame number (PFN); the offset is unchanged (it's the same
byte within the page).

```
physical address = (frame for VPN) * page_size + offset
```

## Page tables

A **page table** maps each VPN to a PFN plus flags (valid/present, read/write/execute, dirty,
accessed). The MMU consults it on every access. Each entry (PTE) says either "this page is in
frame X" or "not present" (triggering a page fault).

A single flat table for a 48-bit address space would be enormous, so x86-64 uses a **multi-level
(4-level)** page table — a tree. The virtual address is sliced into several index fields, one per
level, plus the offset:

```
| L4 idx | L3 idx | L2 idx | L1 idx | offset |
```

Walking the tree (one memory access per level) yields the frame. Multi-level tables are sparse:
unused regions need no lower-level tables, saving huge amounts of space.

## A worked translation

Suppose 4 KiB pages and a (simplified) single-level table. Virtual address `0x5004`:

- offset = low 12 bits = `0x004` = 4
- VPN = `0x5004 >> 12` = `0x5` = 5

If the page table says VPN 5 → frame 9, then:

- physical = `9 * 4096 + 4` = `36868` = `0x9004`.

The high bits got remapped (5 → 9); the offset stayed (4).

## The TLB

Walking a 4-level page table means up to **4 extra memory accesses per memory access** — far too
slow. The **Translation Lookaside Buffer (TLB)** is a small, fast cache of recent VPN→PFN
translations inside the MMU. On a **TLB hit**, translation is nearly free; on a **miss**, the
hardware walks the page table (slow) and caches the result.

This is why **locality matters for translation too**: touching many pages with poor locality
causes TLB misses, and huge working sets can thrash the TLB. (Huge pages — 2 MiB / 1 GiB — reduce
TLB pressure by covering more memory per entry.)

## Sizing the levels, and TLB reach

The 4-level split isn't arbitrary. With 4 KiB pages the offset is 12 bits, leaving 36 bits of VPN
split into **four 9-bit indices** — and 2^9 = **512** entries per table, which is exactly one 4 KiB
page of 8-byte entries. So each page table is itself one page. This is why 48-bit virtual addresses
use 4 levels (`12 + 9*4 = 48`); 5-level paging adds one more index for 57-bit spaces.

**TLB reach** = (TLB entries) × (page size). A 64-entry TLB with 4 KiB pages covers only 256 KiB — so
a loop over a few MB thrashes the TLB even if the data fits in L2. **Huge pages** (2 MiB) multiply
reach 512×: the same 64 entries now cover 128 MiB, which is why databases and JVMs enable them for big
heaps.

## Page faults and demand paging

If a PTE is marked **not present**, the access triggers a **page fault** — a trap into the OS. The
OS decides:

- The page is valid but not yet loaded (e.g., from the executable or swapped out) → load it into a
  free frame, update the PTE, and **retry** the instruction. This is **demand paging**: pages are
  brought in only when actually touched.
- The access is illegal (unmapped/permission) → deliver `SIGSEGV`.

Demand paging is why a large program starts quickly: only the pages it actually uses get loaded.

## Recap

- Memory is paged (typically 4 KiB); a virtual address = VPN + 12-bit offset.
- Page tables (multi-level on x86-64) map VPN→frame with permission flags; they're sparse.
- The TLB caches translations so most accesses avoid the slow page-table walk; locality helps it.
- Not-present pages fault; the OS demand-pages them in and retries, or raises SIGSEGV.
