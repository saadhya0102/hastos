# Memory Models and Ordering

Here we reach the deep end of concurrency. On modern hardware, the order in which one thread's
memory writes become **visible** to another thread is **not** necessarily the order you wrote them
in code. The **memory model** defines the rules, and **memory ordering** on atomics is how you
regain control. This underpins all correct lock-free code.

## Why operations get reordered

Both the **compiler** (optimizer) and the **CPU** (out-of-order execution, store buffers) may
reorder memory operations to go faster, as long as a *single thread's* observable behavior is
preserved. But across threads, those reorderings become visible. A write to `data` then `flag` in
your source might be seen by another thread as `flag` first, then `data` — breaking naive
hand-rolled synchronization.

```c
// Thread 1                    // Thread 2
data = 42;                     while (!ready) {}
ready = 1;                     printf("%d\n", data);  // might print 0!
```

Without proper ordering, Thread 2 can observe `ready == 1` *before* it sees `data == 42`.

## Sequential consistency: the intuitive model

**Sequential consistency (SC)** is the easy-to-reason-about model: it's *as if* all threads'
operations were interleaved into one global order consistent with each thread's program order. C11
atomics default to `memory_order_seq_cst`, which gives SC — correct and simple, but potentially the
slowest because it forbids many reorderings (often requiring fences).

```c
atomic_store(&x, 1);              // defaults to seq_cst
int v = atomic_load(&x);
```

If unsure, **use the default (seq_cst)**: correct first, optimize later.

## Acquire/release: cheaper, sufficient ordering

The most useful weaker ordering is **acquire/release**, which creates a one-way barrier perfect for
publishing data:

- A **release** store ensures all writes *before* it in this thread are visible to any thread that
  later does an **acquire** load of the same atomic.
- An **acquire** load ensures reads *after* it see everything the releasing thread wrote before its
  release.

This is exactly the "publish then read" pattern — and it's how a lock works (unlock = release,
lock = acquire):

```c
// Thread 1 (publisher)            // Thread 2 (consumer)
data = 42;                         if (atomic_load_explicit(&ready, memory_order_acquire)) {
atomic_store_explicit(&ready, 1,       // sees data == 42, guaranteed
  memory_order_release);               printf("%d\n", data);
                                   }
```

The release/acquire pair establishes a **happens-before** edge: everything before the release
happens-before everything after the matching acquire.

## Relaxed ordering

`memory_order_relaxed` provides **atomicity but no ordering** — the operation won't tear, but it
imposes no happens-before relationship with other memory. It's correct only for things like
independent counters where you don't need to order other memory around them (e.g., a statistics
counter you read at the end).

```c
atomic_fetch_add_explicit(&hits, 1, memory_order_relaxed);  // count only; ordering irrelevant
```

Using relaxed where you actually need ordering is a classic, devastating bug.

## Hardware models: x86 vs ARM, and fences

The *same* C code can behave differently per CPU because architectures have different **hardware
memory models**:

- **x86-64 is TSO (Total Store Order)** — relatively strong. The only reordering it allows is
  **StoreLoad**: a later load can appear to execute before an earlier store to a *different* address
  (because stores sit in a per-core **store buffer** before draining to cache). Loads aren't reordered
  after loads, etc. So acquire/release are nearly free on x86 (often plain moves), but a full barrier
  (`mfence` / a `lock`-prefixed op) is needed to stop StoreLoad — the reason `seq_cst` costs more.
- **ARM/POWER are weakly ordered** — almost anything can reorder unless you add barriers. Code that
  "happened to work" on x86 often breaks on ARM (e.g., Apple Silicon), exposing missing ordering.

This is why you express ordering with the **portable C11 atomic orderings** (which the compiler lowers
to the right fences per target) instead of assuming a CPU. `atomic_thread_fence(memory_order_seq_cst)`
inserts a standalone barrier when you need one.

## Practical guidance

- **Default to `seq_cst`** until profiling shows it matters.
- Use **acquire/release** for the publish/consume pattern (and it's what lock acquire/release map
  to).
- Use **relaxed** only for genuinely independent atomics (counters, flags you don't order around).
- Most application code should just use **locks**, which give you correct ordering for free.
  Hand-rolled orderings are for performance-critical lock-free code — write tests and run TSan.

<Aside kind="insight">
Mantra: *correctness first.* Reach for the strongest ordering (seq_cst) or just a mutex, prove it
right, and only relax orderings under measurement. Subtle memory-ordering bugs are nearly
impossible to reproduce and debug after the fact.
</Aside>

## Recap

- Compilers and CPUs reorder memory ops; cross-thread visibility can differ from program order.
- Sequential consistency (the C11 default) is the intuitive, correct, sometimes-slow baseline.
- Acquire/release creates a happens-before edge — the cheap, correct tool for publish/consume.
- Relaxed = atomic without ordering (counters only). Prefer locks/seq_cst; relax only with care.
