# Byte Ordering: Endianness

A single byte has no ordering question — but a multi-byte value like a 32-bit integer must be
stored as a sequence of bytes in memory, and there are two conventions for *which byte goes
first*. This is **endianness**, and it bites you whenever bytes cross a boundary: a file, a
network socket, or a reinterpreting cast.

## The two orders

Take the 32-bit value `0x12345678`. Its bytes, from most significant to least, are `12 34 56
78`. How are they laid out starting at some address `A`?

```
Little-endian (x86, ARM default):     Big-endian (network order, older RISC):
  A+0: 78                               A+0: 12
  A+1: 56                               A+1: 34
  A+2: 34                               A+2: 56
  A+3: 12                               A+3: 78
```

- **Little-endian**: the *least* significant byte comes first (lowest address). x86-64 is
  little-endian.
- **Big-endian**: the *most* significant byte comes first. This is also **network byte order**.

<Aside kind="insight">
Mnemonic: *little*-endian stores the *little* (least significant) end first. Within a byte,
bit order isn't affected — endianness is only about the order of whole bytes.
</Aside>

## Detecting endianness at runtime

Because casting a pointer lets you view an `int` as bytes, you can check at runtime which byte
comes first:

<MiniIDE language="c" starter={`#include <stdio.h>
#include <stdint.h>
int main(void) {
  uint32_t x = 0x12345678u;
  unsigned char *p = (unsigned char *)&x;  /* view x as raw bytes */
  printf("first byte = 0x%02X\\n", p[0]);
  if (p[0] == 0x78) puts("little-endian");
  else if (p[0] == 0x12) puts("big-endian");
  return 0;
}`} />

This trick — aliasing an integer through an `unsigned char *` — is one of the few pointer
re-interpretations that C explicitly allows (`unsigned char` can alias anything).

## Swapping byte order

To convert between orders, reverse the bytes. For a 32-bit value:

```c
uint32_t bswap32(uint32_t x) {
  return ((x & 0x000000FFu) << 24) |
         ((x & 0x0000FF00u) << 8)  |
         ((x & 0x00FF0000u) >> 8)  |
         ((x & 0xFF000000u) >> 24);
}
```

Each byte is masked out, shifted to its mirrored position, and OR'd back together — a direct
application of the masking skills from the bit-ops lesson.

## Network byte order

Network protocols standardized on **big-endian** ("network byte order") so machines of
different architectures can interoperate. The POSIX helpers convert between host and network
order:

- `htonl` / `htons` — host to network (long / short)
- `ntohl` / `ntohs` — network to host

On a big-endian host these are no-ops; on little-endian x86 they byte-swap. Always convert
multi-byte fields when reading/writing the wire.

## Why it matters

- **Binary files & formats.** A file written little-endian and read big-endian yields garbage
  unless you account for it. Formats specify an endianness (or include a byte-order mark).
- **Networking.** Forgetting `htons` on a port number is a classic bug — the port ends up
  byte-swapped.
- **Casting buffers to structs.** Reinterpreting a received byte buffer as a struct of ints
  only works if the sender's endianness (and alignment/padding) match.

## Recap

- Endianness is the order of *bytes* within a multi-byte value.
- Little-endian (x86) stores the least significant byte first; big-endian stores the most
  significant first.
- You can detect endianness by aliasing an int through `unsigned char *`.
- Byte-swap with mask-and-shift; network byte order is big-endian (`htonl`/`ntohl`).
- Mismatched endianness corrupts files, packets, and reinterpreted buffers.
