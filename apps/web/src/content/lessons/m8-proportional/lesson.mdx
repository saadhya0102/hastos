# Proportional-Share Scheduling

Instead of optimizing turnaround or response, **proportional-share** schedulers aim for
**fairness**: each job should get a CPU share proportional to its assigned weight. This model
underlies modern fair schedulers and resource-control systems (cgroups, container CPU limits, VM
vCPU shares).

## Lottery scheduling

Give each job some number of **tickets** proportional to its desired share. At each scheduling
decision, **draw a random ticket**; the job holding it runs the next slice. Over time, a job with
more tickets wins proportionally more often, so it gets proportionally more CPU.

```
A holds 75 tickets, B holds 25 (total 100).
Each tick: random number in [0,100); if < 75, run A, else run B.
Over many ticks, A runs ~75% of the time, B ~25%.
```

Lottery is simple, requires no global state, and degrades gracefully — but it's **probabilistic**, so
short runs can deviate from the exact ratio (the law of large numbers only guarantees fairness *over
many draws*).

### Ticket mechanisms

Lottery has some elegant tricks that make it composable:

- **Ticket currency**: a user can hand out tickets in their own "currency" to their jobs; the system
  converts each user's currency to global tickets by their global allocation. This lets users
  subdivide *their* share without affecting others.
- **Ticket transfer**: a client can temporarily hand its tickets to a server working on its behalf
  (helps avoid priority-inversion-like effects).
- **Ticket inflation**: in a cooperative (mutually trusting) group, a job can temporarily create more
  tickets to grab CPU when it needs a burst — no communication required.

## Stride scheduling

**Stride** makes proportional share **deterministic**. Each job has a **stride** inversely
proportional to its tickets (`stride = BIG / tickets`) and a running **pass** value. Always run the
job with the **lowest pass**, then add its stride to its pass. More tickets → smaller stride → pass
grows slower → runs more often. No randomness, exact proportions.

```
BIG = 100.  A: tickets 100 -> stride 1;  B: tickets 50 -> stride 2

pass:      A   B     who runs (lowest pass), then advance its pass
t0:        0   0     A (tie->A);  A.pass += 1
t1:        1   0     B;           B.pass += 2
t2:        1   2     A;           A.pass += 1
t3:        2   2     A;           A.pass += 1
t4:        3   2     B;           B.pass += 2
...        A runs ~2x as often as B, exactly.
```

The catch stride has that lottery avoids: **global per-job state** (the pass values). A subtle issue
is a job that **joins late** — if you set its pass to 0 it would monopolize the CPU until it catches
up. The fix: set a new job's pass to the current **minimum pass** (or global pass), so it starts
"even" with everyone else. CFS uses exactly this idea.

## Linux CFS: proportional share in practice

Modern Linux uses the **Completely Fair Scheduler (CFS)**, essentially stride's idea generalized and
made efficient:

- Each task accumulates **virtual runtime (`vruntime`)** as it runs. CFS always runs the task with
  the **smallest `vruntime`** (it's fallen furthest behind) — the stride "lowest pass" rule.
- **Weights from `nice`**: a task's `nice` value (−20…+19) maps to a weight; `vruntime` advances
  *inversely* to weight, so higher-priority (more weight) tasks accumulate `vruntime` more slowly and
  thus run more. Each nice level is ~1.25× CPU, so a difference of ~10 nice ≈ 10× the CPU share.
- **Efficiency**: ready tasks live in a **red-black tree** keyed by `vruntime`, so "pick the min" and
  reinsert are `O(log n)`.
- **Tuning without fixed quanta**: `sched_latency` is the target period in which every runnable task
  should run once; each task's slice ≈ `sched_latency × (weight / total_weight)`. A
  `min_granularity` floor prevents slices from getting so tiny that context-switch overhead
  dominates when there are many tasks. New/waking tasks get `vruntime` set near `min_vruntime` so
  they don't unfairly surge (the "join late" fix again).

<Aside kind="insight">
Lottery vs. stride mirrors a common engineering choice: randomized (simple, stateless, approximate)
vs. deterministic (exact, needs per-job state). CFS chose the deterministic path — plus a balanced
tree for scale — because a general-purpose OS wants predictable fairness.
</Aside>

## Where proportional share shows up

- **Containers/cgroups**: `cpu.weight` (cgroup v2) is literally a proportional share of CPU among
  cgroups; `cpu.max` adds a hard cap (bandwidth control).
- **Hypervisors**: vCPU "shares"/"weights" divide physical CPUs among VMs.
- **QoS systems**: weighted fair queueing in networking is the same math applied to link bandwidth.

## Recap

- Proportional-share targets fairness: CPU share ∝ assigned weight/tickets.
- **Lottery**: random ticket draw — simple, stateless, probabilistic; supports currency/transfer/
  inflation.
- **Stride**: lowest-pass-runs, advance by stride — deterministic, exact; new jobs start at the min
  pass.
- **CFS** (Linux): smallest-`vruntime`-runs, weights from `nice`, red-black tree, `sched_latency` /
  `min_granularity` tuning — proportional share at production scale.
