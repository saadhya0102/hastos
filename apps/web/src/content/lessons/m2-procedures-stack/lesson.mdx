# Procedures, the Stack, and the ABI

Function calls feel atomic in C, but at the machine level they're a careful dance of stack
manipulation and register conventions. This is the **calling convention** (ABI), and it's the
contract that lets separately-compiled functions — even in different languages — call each other.

## The stack

The stack is a region of memory that grows **downward** (toward lower addresses) on x86-64.
`rsp` always points at the current top. Two instructions manage it:

- `push x` — decrement `rsp` by 8, then store `x` at `[rsp]`.
- `pop r`  — load `[rsp]` into `r`, then increment `rsp` by 8.

The stack holds return addresses, saved registers, local variables, and spilled values.

## call and ret

- `call f` — **pushes the return address** (the address of the instruction after the call)
  onto the stack, then jumps to `f`.
- `ret` — **pops the return address** off the stack into the instruction pointer, resuming the
  caller.

So a `call`/`ret` pair is really "push where to come back, jump; later pop it and go back."
The return address living on the stack is exactly what buffer overflows attack (next lesson).

## The System V AMD64 calling convention

On Linux/macOS, integer/pointer arguments go in registers, in order:

```
arg1  arg2  arg3  arg4  arg5  arg6   (further args go on the stack)
rdi   rsi   rdx   rcx   r8    r9
```

The **return value** comes back in `rax` (and `rdx` for 128-bit returns). Floating-point args
use `xmm0`–`xmm7`.

```c
long add3(long a, long b, long c) { return a + b + c; }
```
```
add3:
    lea rax, [rdi + rsi]   ; rax = a + b   (a=rdi, b=rsi)
    add rax, rdx           ; rax = (a+b) + c   (c=rdx)
    ret                    ; return value in rax
```

## Caller-saved vs. callee-saved

Registers are shared, so the ABI splits responsibility for preserving them across a call:

- **Caller-saved** (volatile): `rax, rcx, rdx, rsi, rdi, r8–r11`. A callee may clobber these,
  so if the *caller* needs them after the call, it must save them first.
- **Callee-saved** (non-volatile): `rbx, rbp, r12–r15`. If a *callee* wants to use these, it
  must save and restore them (you'll see `push rbx` ... `pop rbx`).

This is why disassembled functions often start by pushing `rbx`/`r12` and end by popping them.

## Stack frames and the frame pointer

Each active call has a **stack frame** holding its saved registers, locals, and the return
address. A traditional prologue/epilogue uses `rbp` as a stable **frame pointer**:

```
func:
    push rbp            ; save caller's frame pointer
    mov  rbp, rsp       ; establish our frame pointer
    sub  rsp, 32        ; reserve 32 bytes for locals
    ...                 ; body uses [rbp - k] for locals
    leave               ; mov rsp, rbp ; pop rbp   (tear down frame)
    ret
```

Optimized code often **omits the frame pointer** (`-fomit-frame-pointer`), freeing `rbp` as a
general register and addressing locals via `rsp` directly.

<Aside kind="note">
The **red zone** is a 128-byte area just below `rsp` that leaf functions (those that call no one)
may use as scratch without adjusting `rsp` — a small optimization in the System V ABI. Also note
the ABI requires `rsp` to be 16-byte aligned at each `call`.
</Aside>

## Recursion on the stack

Each recursive call gets its own frame, which is why deep recursion can overflow the stack:

```c
long fact(long n) { return n <= 1 ? 1 : n * fact(n - 1); }
```
Every `fact` call pushes a return address and frame; `fact(100000)` would exhaust the stack
(stack overflow) — the machine-level reason recursion depth is bounded.

## Tail calls

If the **last** action of a function is a call whose result it returns directly, the compiler can
reuse the current frame instead of adding a new one — a **tail-call optimization (TCO)**, emitted as a
plain `jmp` rather than `call`. That turns tail recursion into a loop with **O(1)** stack.

```c
long fact(long n) { return n <= 1 ? 1 : n * fact(n-1); }   // NOT a tail call: must multiply AFTER
long fact_acc(long n, long acc) {                          // tail call: return is the call itself
  return n <= 1 ? acc : fact_acc(n-1, n*acc);
}
```

`fact` can't be TCO'd (work remains after the recursive call — the multiply), so it grows the stack;
`fact_acc` can become a `jmp` loop. This is why rewriting with an accumulator avoids stack overflow.
(C compilers do TCO at `-O2` but don't *guarantee* it; languages like Scheme require it.)

## Returning large values

Return values that don't fit in `rax:rdx` (e.g., a big struct) use a hidden convention: the caller
allocates space and passes a **pointer to it as a hidden first argument** in `rdi`, shifting the real
arguments to `rsi, rdx, ...`. The callee writes the result through that pointer and returns the
pointer in `rax`. This is why "returning a struct by value" isn't as free as it looks.

## Recap

- The stack grows down; `push`/`pop` move `rsp`; `call` pushes a return address, `ret` pops it.
- System V passes args in `rdi, rsi, rdx, rcx, r8, r9`, returns in `rax`.
- Caller-saved (`rax, rcx, rdx, rsi, rdi, r8–r11`) vs callee-saved (`rbx, rbp, r12–r15`).
- Frames hold locals/saved regs; `rbp` is the optional frame pointer; mind the red zone and
  16-byte alignment.
