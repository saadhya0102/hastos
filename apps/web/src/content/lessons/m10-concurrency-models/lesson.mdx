# Server Concurrency Models

A real server handles **many clients at once**. How it does so — the concurrency model — determines
its scalability. This ties together processes (M5), threads (M7), and I/O, and explains how modern
high-performance servers reach hundreds of thousands of connections.

## Process-per-connection

The oldest model: `accept()` a connection, then `fork()` a child to handle it. Simple and isolated
(a crash in one child doesn't affect others), but **heavyweight** — each process has its own
address space; thousands of connections mean thousands of processes, which is memory-hungry and
context-switch-heavy.

## Thread-per-connection

Spawn a **thread** per connection instead of a process. Cheaper than processes (shared address
space), and easy to write with blocking I/O (each thread just `read`/`write`s its socket). But it
still doesn't scale to *hundreds of thousands* of connections — thread stacks and scheduler
overhead add up, and shared state needs synchronization. A **thread pool** (M7) bounds the cost.

## Event-driven (the event loop)

The scalable model: a **single thread** (or a few) manages **many** connections using
**non-blocking I/O** and a readiness API. The OS tells you *which* sockets are ready to read/write,
and you handle just those, never blocking on any one. This is the **event loop**:

```
register all sockets with epoll
loop:
  events = epoll_wait()        // sleep until some sockets are ready
  for each ready socket:
    do a non-blocking read/write and advance that connection's state
```

One thread can juggle tens of thousands of connections because it only does work for sockets that
are actually ready, and never stalls waiting on a slow client. This is how nginx, Redis, and Node
achieve their concurrency.

## epoll vs. select, and C10k

The readiness API matters at scale:

- **`select`/`poll`**: you pass the *entire* set of fds every call, and the kernel scans them all —
  **O(n)** per call. Fine for a few fds, terrible for many.
- **`epoll`** (Linux) / **kqueue** (BSD): you register interest once; the kernel returns only the
  ready fds — effectively **O(1)** in the number of idle connections. This is what made solving the
  **C10k problem** (10,000 concurrent connections on one machine) practical.

## Blocking vs. non-blocking I/O

- **Blocking**: a `read` sleeps until data arrives. Simple, but a thread can serve only one
  connection at a time while blocked — fine for thread-per-connection.
- **Non-blocking**: `read` returns immediately (with `EAGAIN` if no data), letting one thread
  manage many sockets via the event loop. The basis of event-driven servers.

Modern designs often combine them: an event loop per core (e.g., several threads each running their
own epoll loop), getting both multicore parallelism and per-thread efficiency.

## Recap

- Process-per-connection: isolated but heavy; thread-per-connection: lighter, still limited at huge
  scale (bound it with a pool).
- Event-driven (one thread + non-blocking I/O + epoll) scales to hundreds of thousands of
  connections by servicing only ready sockets.
- epoll/kqueue beat select/poll (O(1) vs O(n)) — the key to solving C10k.
- Blocking I/O suits threads; non-blocking I/O powers event loops; real servers often run an event
  loop per core.
