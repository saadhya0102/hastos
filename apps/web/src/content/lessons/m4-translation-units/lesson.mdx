# Translation Units, Declarations, and Definitions

Before a program runs, separately-written source files must be welded into one executable. That
job belongs to the **linker**, and to understand it you first need a precise grip on three ideas
that beginners blur together: the **translation unit**, the **declaration**, and the
**definition**.

## The translation unit

A **translation unit** is what the compiler actually compiles: a single `.c` file *after the
preprocessor has run* (all `#include`s pasted in, all macros expanded). Crucially, **the compiler
sees one translation unit at a time** — it has no idea what's in your other `.c` files.

This is the single most important fact about C compilation. When `main.c` calls a function
defined in `util.c`, the compiler processing `main.c` never sees `util.c`. It only needs a
*promise* that the function exists; the linker connects the promise to the real thing later.

## Declarations vs. definitions

- A **declaration** introduces a name and its type — a promise that it exists *somewhere*.
  `int square(int);` declares `square`. `extern int g;` declares a global `g`. Declarations
  produce **no code or storage**.
- A **definition** actually creates the thing: the function body or the variable's storage.
  `int square(int x) { return x*x; }` defines `square`; `int g = 0;` defines `g`.

You may **declare** a name as many times as you like, but you must **define** it exactly once.
This is the **One Definition Rule (ODR)**.

```c
int square(int);              // declaration (a promise)
int square(int x){return x*x;} // definition (the real thing)
```

## Why headers and .c files split this way

The convention falls directly out of the rules above:

- **Headers (`.h`)** hold **declarations** — function prototypes, `extern` globals, types,
  macros. They're `#include`d into many translation units so everyone shares the same promises.
- **Source files (`.c`)** hold **definitions** — the actual bodies and storage, compiled exactly
  once.

If you accidentally put a *definition* in a header that's included by multiple `.c` files, each
translation unit gets its own definition, and the linker complains about **multiple definitions**.

<Aside kind="pitfall">
This is why you write `extern int counter;` in a header and `int counter = 0;` in exactly one
`.c` file. Putting `int counter = 0;` in the header would define it in every file that includes
it — a multiple-definition link error.
</Aside>

## Compile-time vs. link-time errors (revisited precisely)

Now the distinction is sharp:

- A **compile error** happens within one translation unit — bad syntax, type mismatch, using an
  undeclared name. The compiler can detect it because everything it needs is in that unit (given
  the declarations from headers).
- A **link error** happens when combining units — a declared-but-never-defined symbol
  (`undefined reference`) or a symbol defined in two units (`multiple definition`). The compiler
  can't catch these because no single unit has the whole picture.

```c
// main.c
int helper(int);          // declared -> compiles fine
int main(void){ return helper(3); }
// (no helper definition anywhere)  -> LINK error: undefined reference to 'helper'
```

## Recap

- A translation unit is one preprocessed `.c` file; the compiler sees only one at a time.
- Declarations promise; definitions create. Declare freely, define once (the ODR).
- Headers carry declarations across units; `.c` files carry definitions.
- Compile errors are within a unit; link errors (undefined/multiple definition) arise when
  joining units.
