# The Client-Server Model

Networking lets programs on different machines communicate. The dominant pattern is
**client-server**: a long-running **server** waits for requests; **clients** initiate connections
and ask for work. The web, databases, and most distributed systems are built on it.

## Roles

- A **server** binds to a known address/port and **listens**, accepting incoming connections and
  responding to requests. It's reactive — it waits.
- A **client** knows the server's address and **initiates** a connection, sends requests, reads
  responses. It's proactive.

The same machine can be both (e.g., a web server is a client to its database). "Client" and
"server" describe *roles in an interaction*, not fixed machine types.

## Addresses, ports, and the stack

- An **IP address** identifies a host on the network (e.g., `93.184.216.34`, or an IPv6 address).
- A **port** (0–65535) identifies a specific service/socket on that host (HTTP = 80, HTTPS = 443).
- The **protocol stack** layers: application (HTTP) over transport (TCP/UDP) over network (IP) over
  link (Ethernet/Wi-Fi). Each layer wraps the one above with its own header.

So "connect to example.com:443" means: resolve the name to an IP, then open a transport connection
to port 443 on that host.

## Identifying a connection: the 5-tuple

A single server port (say 443) handles thousands of simultaneous clients. How does the OS keep
them separate? Each connection is uniquely identified by a **5-tuple**:

```
(protocol, source IP, source port, destination IP, destination port)
```

Two clients hitting the same server port differ in their source IP/port, so their connections are
distinct. This is why one listening socket can fan out to many connected sockets.

## Encapsulation: how the layers wrap data

Each layer **wraps** the layer above it with its own header (and strips it on the way up) —
**encapsulation**. Your HTTP bytes become a TCP segment, inside an IP packet, inside an Ethernet
frame:

```
[ Ethernet [ IP [ TCP [ HTTP payload ] ] ] ]
   link       net   transport  application
```

At the destination, each layer peels its header and hands the rest up. This layering is why you can
swap Wi-Fi for Ethernet (different link layer) without changing your HTTP code, and why a router only
needs to look at the IP header, not your application data.

## Names, DNS, and NAT

Two pieces make the internet usable at human scale:

- **DNS** translates a name like `example.com` into an IP address (a distributed, cached hierarchy of
  name servers). "Connect to example.com" first does a DNS lookup, then connects to the returned IP.
- **NAT (Network Address Translation)**: home/office networks use **private** IPs behind one public
  IP; the router rewrites source IP/port on the way out and tracks the mapping so replies return to
  the right internal host. This is a big reason **source ports** matter (they disambiguate many
  internal clients sharing one public IP) — and why inbound connections to machines behind NAT need
  port forwarding or a tunnel (recall the Piston setup!).

## Recap

- Client-server: servers listen/accept; clients initiate. Roles, not machine types.
- IP addresses identify hosts; ports identify services; protocols layer (HTTP/TCP/IP/link).
- Connections are demultiplexed by the 5-tuple, so one port serves many clients.
