# Key-Value Stores and Indexing

At the bottom of every database is a **storage engine** that maps **keys to values** on durable
storage. Understanding how it indexes data — and the amplification trade-offs — explains why
databases choose B-trees or LSM-trees, and prepares you to implement both.

## The key-value model

The core API is tiny:

- `put(key, value)` — store/overwrite.
- `get(key)` — retrieve.
- `delete(key)` — remove.
- often `scan(range)` — iterate keys in order.

Everything else (SQL, documents, secondary indexes) is built atop this. The hard part isn't the
API; it's doing it **fast and durably** over slow storage.

## Why an index

The naive store — an append-only log of `(key, value)` records — makes `put` trivially fast
(append) but `get` O(n) (scan the whole log). An **index** is an auxiliary data structure that maps
a key to the **location** of its value, turning `get` into O(log n) or O(1). But the index must be
**maintained on every write**, so it trades write cost for read speed. The whole field is about
choosing that trade-off well.

## The simplest real index: a hash index

Before trees, consider the simplest index that actually ships (Bitcask, used by Riak): keep the
append-only log on disk, plus an **in-memory hash map** from each key to the **byte offset** of its
latest value in the log.

```
put(k,v):  append (k,v) to the log; hash[k] = offset_of_that_record
get(k):    look up hash[k] -> seek to that offset -> read the value
```

`put` and `get` are both ~O(1), and durability comes for free from the append log. But the design has
sharp limits that motivate trees:

- **All keys must fit in RAM** (the hash map is in memory).
- **No efficient range scans / ordered iteration** (a hash has no order).
- The log grows forever, so you need **compaction**: periodically rewrite live records into new
  **segments** and discard stale ones (an overwritten or deleted key's old records are garbage).

## Sorted files (SSTables) fix ordering

If instead we keep each on-disk segment **sorted by key** (a Sorted String Table), we gain range
scans, a sparse in-memory index (one entry per block, not per key — so keys needn't all fit in RAM),
and simple **merge** compaction of sorted segments. That's precisely the foundation LSM-trees build
on, while B-trees keep the sorted data in a paged tree updated in place. Both are "keep it sorted";
they differ in *how they absorb writes*.

## Amplification: the key metrics

Storage engines are judged by three "amplifications":

- **Read amplification** — extra reads per logical read (e.g., checking several files/levels).
- **Write amplification** — extra bytes written per logical write (e.g., rewriting during
  compaction, or B-tree page rewrites).
- **Space amplification** — extra storage used (e.g., obsolete versions awaiting compaction).

No design wins all three; engines pick a point on this triangle. **B-trees** optimize reads;
**LSM-trees** optimize writes. The next lessons examine each.

## Read- vs. write-optimized

- **Read-optimized (B-tree)**: data kept sorted in a balanced on-disk tree; a read is a few page
  fetches. Writes update pages in place (random writes, page rewrites) — higher write cost.
- **Write-optimized (LSM-tree)**: writes are buffered in memory and flushed as sorted runs
  (sequential writes); reads may consult several runs (higher read amplification, mitigated by
  bloom filters), and background **compaction** merges runs.

The right choice depends on your read/write ratio — a recurring systems decision.

## Recap

- A storage engine is a durable key-value map; indexes trade write cost for read speed.
- Designs are judged by read/write/space amplification — you can't minimize all three.
- B-trees are read-optimized (in-place, sorted); LSM-trees are write-optimized (buffer + flush +
  compact).
