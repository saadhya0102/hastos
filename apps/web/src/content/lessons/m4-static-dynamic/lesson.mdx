# Static vs. Dynamic Libraries

Real programs link against libraries — collections of precompiled object files. There are two
flavors, **static** and **dynamic**, with very different consequences for binary size, updates,
and deployment. Understanding both (and the famous link-order gotcha) is everyday systems
knowledge.

## Static libraries (.a archives)

A static library is just an **archive** of `.o` files (think of it as a `.zip` of object files),
created with `ar`:

```
ar rcs libmath.a add.o mul.o
gcc main.o libmath.a -o app      # or: gcc main.o -L. -lmath
```

At link time, the linker pulls **only the object files it needs** out of the archive to resolve
undefined symbols, and copies that code **into the executable**. The library isn't needed at
runtime — everything is baked in.

### The link-order gotcha
The linker processes files **left to right**, carrying a set of currently-unresolved symbols. It
only pulls an object from an archive if that archive appears **after** the file that needs it.
So this fails:

```
gcc -lmath main.o      # WRONG: -lmath seen first, main.o's needs not known yet
gcc main.o -lmath      # RIGHT: main.o creates the need, then -lmath satisfies it
```

This trips up nearly everyone at least once: "undefined reference" even though the library is
right there — because it was listed too early.

## Dynamic / shared libraries (.so)

A shared library is loaded at **runtime**, not copied into the executable:

```
gcc -shared -fPIC add.c mul.c -o libmath.so
gcc main.o -L. -lmath -o app          # records a DEPENDENCY on libmath.so
```

The executable stores a *reference* ("I need `libmath.so` and symbol `add`"), and the **dynamic
linker/loader** (`ld.so`) resolves it when the program starts (or lazily on first use). Many
programs share **one copy** of the library in memory.

## Load-time vs. run-time linking

- **Load-time dynamic linking** — the loader resolves the library's symbols when the program
  launches (the common case). Lazy binding via the PLT defers each function's resolution to its
  first call.
- **Run-time dynamic linking** — the program explicitly loads a library *while running* with
  `dlopen`/`dlsym`. This powers plugins: load code chosen at runtime.

```c
void *h = dlopen("./plugin.so", RTLD_NOW);
int (*f)(int) = dlsym(h, "process");   // grab a function by name at runtime
```

## Trade-offs

| Aspect | Static (.a) | Dynamic (.so) |
| --- | --- | --- |
| Binary size | Larger (code copied in) | Smaller (code shared) |
| Memory across processes | Each has its own copy | One shared copy |
| Updates / security fixes | Must relink & redeploy | Update the .so once; all users benefit |
| Startup | Slightly faster (no resolution) | Slight load-time resolution cost |
| Deployment | Self-contained, portable | Must ship/find the right .so versions |

The update story is why OS distributions favor shared libraries: a single patched `libssl.so`
fixes every program. The portability story is why container/static builds (e.g., Go binaries,
musl-static C) favor static linking.

## Finding the .so at runtime: sonames and search paths

Dynamic linking adds a deployment question static linking doesn't have: *which* `.so`, and *where*?

- **soname**: a shared library records a versioned name like `libssl.so.3`. The executable records a
  dependency on that soname, so an incompatible `libssl.so.4` won't be picked up by accident. A dev
  symlink `libssl.so` → `libssl.so.3` is used at link time; the soname is used at run time.
- **Search order** for `ld.so`: `DT_RPATH` (legacy) → `LD_LIBRARY_PATH` (env, handy for dev, unsafe
  for setuid) → `DT_RUNPATH` (baked into the binary) → the `ldconfig` cache (`/etc/ld.so.cache`) →
  default dirs (`/lib`, `/usr/lib`). You can inspect what a binary will load with `ldd ./app`.
- **"DLL hell" / diamond problems**: if two dependencies need different, incompatible versions of the
  same library, dynamic linking can't satisfy both — the flip side of the "update one file" benefit.

## Security angle

Shared, position-independent libraries are what make **ASLR** (address-space layout randomization)
possible — the loader can place `libc` at a random address each run, so an attacker can't hardcode
gadget addresses. Modern toolchains build executables as **PIE** (position-independent executables)
for the same reason (next lesson). Fully static binaries trade some of this flexibility for
self-containment.

## Recap

- Static `.a` = archive of objects; needed members are copied into the executable; mind link
  order (libraries after the files that use them).
- Dynamic `.so` = loaded at runtime, shared across processes; resolved by `ld.so` (lazily via the
  PLT), or explicitly via `dlopen`/`dlsym`.
- Trade-offs: static = portable/self-contained; dynamic = smaller, shareable, centrally
  updatable.
