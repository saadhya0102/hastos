# Acing Systems Interviews

Systems and infrastructure interviews differ from generic algorithm rounds: they probe whether you
truly understand the machine, memory, and concurrency — and whether you can *implement* the
canonical artifacts under pressure while reasoning about trade-offs. This lesson is your playbook,
plus a map of the problems in this course to practice.

## The canonical implementation problems

If you can implement these from scratch and discuss their trade-offs, you're ready for most
systems interviews. Practice them in the **Interview Track** and the IDE:

- **Concurrency** (the highest-signal area): spinlock, ticket lock, atomic counter, blocking bounded
  queue, thread pool, **lock-free Treiber stack**, **lock-free MPSC queue**. (Module M7)
- **Memory**: implement an **arena** and a **pool/slab allocator**; discuss fragmentation and
  alignment. (Module M6)
- **Data structures for systems**: **LRU cache** (O(1)), ring buffer, B-tree node search, hash map.
  (Modules M1, M9, M11)
- **OS / storage**: scheduler sims (RR), CLOCK page replacement, LSM merge, WAL/crash consistency.
  (Modules M8, M9, M11)
- **Networking**: HTTP request parser, internet checksum, server concurrency models. (Module M10)

## A structure for solving them

Don't just start typing. Use a repeatable flow:

1. **Clarify** the contract: inputs, outputs, concurrency assumptions (single vs multi consumer?),
   constraints, and what "correct" means. State your assumptions out loud.
2. **Design** before coding: pick the data structure and algorithm, sketch the invariants, and call
   out the complexity. Mention the trade-off you're choosing and why.
3. **Implement** cleanly: handle the edge cases (empty, full, n=0, overflow), and keep functions
   small.
4. **Test**: walk through examples, especially boundary and concurrent interleavings. Name how
   you'd stress-test (TSan/ASan, multiple seeds).
5. **Discuss trade-offs**: complexity, memory, contention, failure modes, and how you'd scale or
   harden it.

## Communicate like a senior engineer

The implementation is table stakes; the **discussion** is what levels you up:

- State **complexity** (time/space) and why it's acceptable.
- Name the **trade-off** you took (e.g., "spinlock for tiny critical sections; a mutex if contended").
- Call out **failure modes** and **concurrency hazards** by name: data race, deadlock, ABA, false
  sharing, lost wakeup, write amplification.
- Mention **how you'd test** it — bounded checks, sanitizers, stress with many seeds.

Interviewers probe with follow-ups; in this course every problem ships **follow-up questions** for
exactly this. Rehearse answering them aloud.

## The other round: systems design

Many systems roles add a **design** round ("design a URL shortener / rate limiter / news feed"). It
rewards the same knowledge at a higher altitude:

- **Estimate first** (the latency ladder from M3): QPS, storage, bandwidth. "1M writes/day ≈ ~12/s
  average, plan for ~100/s peak" frames everything.
- **Reach for the recurring building blocks**: a **cache** (LRU/TTL) in front of storage; **sharding**
  to scale writes; **replication** for read scale + availability; a **queue** for backpressure and
  async work; a **load balancer** up front.
- **Name the trade-offs**: **CAP** (under a partition, choose consistency or availability),
  read-vs-write optimization (B-tree vs LSM from M11), strong vs eventual consistency, sync vs async.
- **Address failure**: what happens when a node dies, a queue backs up, or a cache goes cold
  (thundering herd → add jitter/request coalescing).

Notice these are the *same* concepts you implemented — an LRU cache, a bounded queue with
backpressure, a storage engine's amplification trade-offs — just composed at system scale.

## A rapid-fire fact checklist

Keep these interview-ready (all covered in the course, and in the loading-screen trivia):

- Cache line ~**64 B**; page ~**4 KB**; L1 hit ~**1 ns**, DRAM ~**100 ns**, SSD ~**100 µs**.
- **ABA** problem and fixes (tagged pointers / hazard pointers / RCU).
- **False sharing** → pad to a cache line; **MESI** ping-pong.
- Memory ordering: **seq_cst** default, **acquire/release** for publish/consume.
- **B-tree** (read-optimized) vs **LSM** (write-optimized); read/write/space **amplification**.
- **fork** = COW; **syscall** = trap; a lock's unlock/lock = release/acquire.

## How to use HastOS to prepare

- Work the **Interview Track** end-to-end; for each problem, implement it, then answer its
  follow-ups out loud.
- Use the **Mock Interview** (timed coding + trivia rounds) to simulate pressure.
- When stuck, use **SLAVA** at the *nudge* level first — interviews reward independent problem
  solving; escalate hints sparingly.
- Use the **Review** queue and loading-screen **trivia** to keep the high-value facts (cache line
  size, page size, memory ordering, ABA, amplification) sharp.

## Recap

- Master the canonical implementations — especially concurrency (lock-free queue, spinlock, thread
  pool), allocators, and the LRU cache.
- Solve with a structure: clarify → design → implement → test → discuss trade-offs.
- Communicate complexity, trade-offs, named hazards, and testing — that's the senior signal.
- Practice with the Interview Track, Mock Interview, SLAVA nudges, and spaced review.
