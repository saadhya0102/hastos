# CPU Scheduling: FIFO, SJF, and Round Robin

The scheduler decides **which ready process runs next**. Different policies optimize different
metrics, and the trade-offs are a favorite interview and exam topic. We'll build up from the
simplest policy to the one your laptop actually uses ideas from.

## Two metrics

- **Turnaround time** = completion time − arrival time. How long from submission to done. Good for
  **batch** throughput.
- **Response time** = first-run time − arrival time. How long until the job *first* gets the CPU.
  Good for **interactivity** (you want a keystroke handled quickly).

These often conflict: optimizing turnaround can hurt response time and vice versa.

## FIFO (First-In, First-Out)

Run jobs in arrival order, to completion, non-preemptively. Simple and fair-ish, but suffers the
**convoy effect**: a long job at the front makes short jobs behind it wait, inflating average
turnaround.

```
Jobs (arrive ~0): A=100, B=10, C=10
FIFO order A,B,C: completions 100, 110, 120 -> avg turnaround = (100+110+120)/3 = 110
```

If B and C had gone first, the average would plummet — motivating SJF.

## SJF / STCF (Shortest Job First / Shortest Time-to-Completion First)

**SJF** runs the shortest job first (non-preemptive). For jobs arriving together, SJF is
**optimal for average turnaround**. **STCF** is the preemptive version: if a shorter job arrives,
preempt the running one.

```
A=100, B=10, C=10 with SJF order B,C,A: completions 10, 20, 120 -> avg = (10+20+120)/3 = 50
```

The catch: it needs to **know job lengths** in advance (rarely true), and long jobs can **starve**
if short ones keep arriving.

## Round Robin (RR)

For good **response time**, RR gives each job a small **time quantum**, cycling through the ready
queue. No job waits long for its first turn, so response time is excellent — at the cost of more
context switches and worse average turnaround (every job finishes later because they share).

```
A=B=C=5, quantum=1: they interleave; all finish near the end -> good response, worse turnaround
```

The **quantum trade-off**: too small → responsive but high context-switch overhead; too large →
degrades toward FIFO (poor response). Real systems pick a few milliseconds.

## No single winner

- **FIFO**: simple; convoy effect.
- **SJF/STCF**: optimal turnaround; needs job lengths; can starve long jobs.
- **RR**: great response; worse turnaround; quantum tuning.

Real schedulers can't know job lengths and must balance both metrics — leading to **MLFQ**, next.
