# Producer/Consumer and Thread Pools

The **producer/consumer** pattern — producers put work into a shared queue, consumers take it out
— is the backbone of concurrent systems. Generalize it and you get a **thread pool**: a fixed set
of worker threads pulling tasks from a queue. This is how web servers, job systems, and runtimes
get parallelism without the cost of spawning a thread per task.

## Producer/consumer recap

Producers and consumers communicate through a **bounded queue** (last lesson). This **decouples**
their rates: a burst of production is buffered; consumers drain at their own pace. The queue's
blocking behavior provides natural **backpressure** — producers block when it's full, so a fast
producer can't overwhelm slow consumers or exhaust memory.

## Thread pools

Spawning a new thread per task is expensive (creation cost, unbounded thread counts thrash the
scheduler). A **thread pool** creates `W` worker threads once; each loops: take a task from a
shared queue, run it, repeat. Submitting work just enqueues a task.

```c
// each worker:
for (;;) {
  task = queue_take(pool->queue);   // blocks until a task is available
  if (task.is_shutdown) break;
  task.fn(task.arg);                // run the work
}
```

Benefits: bounded concurrency (exactly `W` tasks run at once), amortized thread cost, and a simple
submit API. This is the engine behind most server request handling.

## Graceful shutdown

Stopping a pool correctly is subtly tricky. You must let in-flight and queued tasks finish (or
explicitly drain), then stop workers without leaving any blocked forever in `queue_take`. Common
approaches:

- A **shutdown flag** plus `broadcast` to wake all workers; workers exit when the flag is set and
  the queue is empty.
- **Poison-pill** tasks: enqueue one special "stop" task per worker; each worker that sees it
  exits. (Simple and avoids missed wakeups.)

Then `join` all workers so resources are released and no thread outlives the pool.

## Scaling up: work-stealing and futures

A single shared queue is simple but becomes a **contention bottleneck** — every worker locks the same
queue on every task. High-performance runtimes (Go's scheduler, Rust's Rayon, Java's ForkJoinPool)
use **work-stealing**: each worker has its **own deque** of tasks, pushing/popping from one end
lock-free-ish; when a worker runs dry, it **steals** from the *other* end of a random victim's deque.
Local work stays local (great cache behavior, no contention); stealing only happens when a worker is
idle, balancing load automatically.

To get a task's **result** back, submission returns a **future/promise**: a handle the caller can wait
on, which the worker fulfills when the task completes (implemented with a per-future
mutex+condvar/flag). This turns "fire and forget" into "submit and later await."

## Sizing the pool

How many workers?

- **CPU-bound** tasks: about **one worker per core** — more just adds context-switch overhead with
  no extra parallelism.
- **I/O-bound** tasks: **more workers than cores** can help, since workers spend time blocked on
  I/O and others can run meanwhile. (Or use async/event-driven I/O instead.)

Oversizing wastes memory and causes scheduler thrash; undersizing leaves cores idle.

## Recap

- Producer/consumer via a bounded queue decouples rates and gives backpressure.
- A thread pool runs `W` reusable workers pulling tasks from a shared queue — bounded concurrency,
  amortized cost.
- Shut down by waking all workers (flag+broadcast or poison pills) and joining them.
- Size ~1 worker/core for CPU-bound work; more for I/O-bound.
