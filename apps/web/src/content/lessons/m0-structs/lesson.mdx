# Structs, Unions, and Alignment

Structs group related data, but how the compiler lays them out in memory — with **alignment** and
**padding** — surprises many people and directly affects memory usage and cache behavior. Unions
overlay fields in the same bytes. Understanding both is essential for systems code, wire formats,
and performance.

## Struct layout and alignment

Each type has an **alignment requirement**: an object of that type must start at an address that's a
multiple of its alignment (usually equal to its size for scalars — `int` aligns to 4, `double`/
pointers to 8). The compiler inserts **padding** bytes between fields so each field is properly
aligned, and pads the whole struct so arrays of it stay aligned.

```c
struct A {
  char  c;   // offset 0
  // 3 bytes padding
  int   x;   // offset 4 (must be 4-aligned)
  char  d;   // offset 8
  // 3 bytes padding (so sizeof is a multiple of 4)
};             // sizeof(struct A) == 12, not 6!
```

The naive sum of field sizes (1+4+1 = 6) is wrong; padding makes it **12**.

## Why padding exists

Hardware accesses aligned data more efficiently (and some architectures fault on misaligned
access). Rather than risk slow or illegal unaligned reads, the compiler aligns every field. The
trailing padding ensures that in an array `struct A arr[2]`, `arr[1]` is also aligned.

## Reordering to shrink structs

Because padding depends on field order, **reordering fields from largest to smallest alignment**
often shrinks a struct:

```c
struct Bad  { char c; int x; char d; };       // 12 bytes (padding between)
struct Good { int x; char c; char d; };        // 8 bytes (fields packed, 2 trailing pad)
```

For hot data structures, this "struct packing" reduces memory and improves cache density — a cheap,
real optimization.

## Unions: overlapping fields

A **union** stores all its members in the **same** bytes — its size is that of its largest member,
and only one member is valid at a time. Unions are used for tagged variants and for reinterpreting
bytes.

```c
union U { int i; float f; unsigned char bytes[4]; };  // sizeof == 4; i, f, bytes share storage
```

Writing one member and reading another is **type punning**. In C it's often allowed (unlike C++),
but reading an *uninitialized* or wrong-sized member is a bug, and it interacts with strict
aliasing. For reliably reinterpreting bytes (e.g., a float's bits), prefer `memcpy` — as you saw in
the IEEE-754 lesson.

## offsetof and the container_of trick

Because layout is fixed and knowable, you can compute a field's byte offset with `offsetof(type,
field)` (from `<stddef.h>`). The famous **`container_of`** macro inverts a pointer: given a pointer to
a *field*, recover a pointer to the *enclosing struct* by subtracting the field's offset. This is how
the Linux kernel's intrusive linked lists work — the list node is embedded in your struct, and
`container_of` gets back to your struct from the node.

```c
#include <stddef.h>
#define container_of(ptr, type, member) \
  ((type *)((char *)(ptr) - offsetof(type, member)))

struct task { int id; struct list_node node; };
// given `struct list_node *n`, recover the task:
struct task *t = container_of(n, struct task, node);
```

## Bitfields and flexible array members

- **Bitfields** pack sub-byte integer fields, handy for hardware registers and compact flags — at
  the cost of implementation-defined ordering, so avoid them for portable wire formats.

```c
struct flags { unsigned ready : 1; unsigned mode : 2; unsigned prio : 5; }; // 1 byte of bits
```

- **Flexible array member (FAM)**: a trailing `type name[]` lets one allocation hold a header plus a
  variable-length payload contiguously — one `malloc`, great locality.

```c
struct packet { size_t len; char data[]; };            // data[] has no size in sizeof
struct packet *p = malloc(sizeof(*p) + n);              // header + n payload bytes, one block
p->len = n;
```

## Practical uses

- **Wire/file formats**: structs must often be packed (`#pragma pack` or manual layout) to match an
  exact byte layout — padding would corrupt the format. Better: serialize field-by-field.
- **Tagged unions**: a struct with an enum tag + a union of variant payloads (a common pattern for
  ASTs, messages, and sum types in C).
- **Cache density**: shrink hot structs (reorder, use smaller int types) so more fit per cache line
  (recall false sharing / SoA from later modules).

## Recap

- Fields are aligned to their type's boundary; the compiler pads between fields and at the end.
- `sizeof` is the sum of fields **plus padding** (and a multiple of the max alignment).
- Reorder fields (largest first) to cut padding and shrink hot structs.
- Unions overlay members in shared bytes (size = largest member); use `memcpy` for safe bit
  reinterpretation.
