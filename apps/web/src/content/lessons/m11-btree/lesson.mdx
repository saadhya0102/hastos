# B-Trees

The **B-tree** is the read-optimized index behind most relational databases and many file systems.
It keeps keys **sorted** in a balanced, shallow tree of disk **pages**, so any key is found in a
handful of page reads. Understanding its search and split operations is essential storage knowledge
and a common interview topic.

## Structure: shallow because of high fanout

A B-tree node is a disk page holding many **sorted keys** and pointers to **children** (one more
child than keys). Because each node has high **fanout** (hundreds of keys per page), the tree is
extremely **shallow**: even billions of keys fit in 3–4 levels.

```
node: [ k0 k1 k2 ... ]  with children c0 c1 c2 c3 ...
      keys in c0 < k0 < keys in c1 < k1 < keys in c2 < ...
```

Shallowness is the whole point: tree height ≈ log_fanout(N), and each level is one slow page read,
so high fanout minimizes disk I/O per lookup.

## Search

Searching is a descent guided by binary search within each node:

1. In the current node, find where the target fits among the sorted keys (binary search).
2. If a key equals the target, done.
3. Otherwise descend into the child between the surrounding keys.
4. Repeat until found or you reach a leaf.

The "where does the target fit" step is exactly a **lower bound** (first key ≥ target), which gives
the child index to follow. You'll implement that core step.

## Insertion with splitting

Inserts keep the tree balanced via **splitting**:

1. Descend to the correct **leaf** and insert the key in sorted order.
2. If the node now exceeds capacity (overfull), **split** it: the median key moves **up** into the
   parent, and the node divides into two half-full nodes.
3. A split can cascade: if the parent overflows, it splits too — possibly up to the root, which
   grows the tree by one level.

Splitting keeps every leaf at the same depth (perfect balance) and every node at least half full,
guaranteeing O(log N) operations.

## B-tree vs B+ tree (what databases actually use)

Textbooks say "B-tree," but nearly every database uses the **B+ tree** variant:

- **All values live in the leaves.** Internal nodes hold *only keys + child pointers* (as routing
  signposts), so they pack more keys per page → even higher fanout → shallower tree.
- **Leaves are linked** in a sorted doubly linked list. A **range scan** (`WHERE age BETWEEN 20 AND
  30`, or `ORDER BY`) descends once to the start key, then walks leaf-to-leaf sequentially — no
  re-descending the tree per row.

This is why B+ trees are the default on-disk index: great point lookups *and* great range/ordered
scans.

## Fanout and height: a quick estimate

Height determines I/O per lookup, and fanout determines height. With 16 KB pages, ~16-byte keys and
~8-byte child pointers, an internal node holds roughly `16384 / 24 ≈ 680` children. Then:

```
height 2:  680^2      ~= 460,000 keys
height 3:  680^3      ~= 314,000,000 keys
height 4:  680^4      ~= 2.1 x 10^11 keys
```

So a handful of levels indexes hundreds of millions of rows — and the top levels stay cached, so a
lookup is often just **one** actual disk read at the leaf.

## Clustered vs secondary indexes

- A **clustered index** stores the table rows *in* the B+ tree leaves in key order (e.g., InnoDB's
  primary key). One index *is* the table.
- A **secondary index** stores the indexed column plus a reference (the primary key or a row id) back
  to the row; a lookup may need a second probe into the clustered index ("bookmark lookup").

## Deletion and rebalancing

Deletes mirror inserts. Remove the key from its leaf; if the node drops **below half full**
(underflow), either **borrow** a key from a sibling (rotation) or **merge** with a sibling, pulling a
separator key down from the parent — which can cascade merges up the tree, occasionally shrinking its
height. In practice many engines defer merges (tolerate partly-empty pages) and reclaim space later,
because deletes are often followed by inserts.

## Concurrency: latch crabbing

Multiple threads walk the tree at once, so engines take short page **latches** while descending using
**crabbing**: latch the child, and once you're sure it won't split/merge (it has room), release the
parent's latch — "crabbing" down hand-over-hand. This keeps concurrency high while preventing a
reader from seeing a node mid-split.

## Why pages, in place

B-trees are tuned for storage: node size = disk/SSD **page** size, so each node read/write is one
I/O. Updates happen **in place** (rewrite the page), which gives great reads and range scans (keys
are sorted and clustered) but causes random writes and write amplification (rewriting full pages
for small changes) — the trade-off versus LSM-trees, next.

## Recap

- B-trees store sorted keys in shallow, high-fanout pages → few page reads per lookup.
- Search = binary-search (lower bound) within a node, then descend; O(log N).
- Insert into a leaf; split overfull nodes, pushing the median up (cascades keep balance).
- Page-sized nodes + in-place updates give fast reads/scans but write amplification.
