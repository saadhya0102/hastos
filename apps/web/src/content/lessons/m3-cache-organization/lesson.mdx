# How Caches Work

A cache is a small, fast memory that holds copies of recently-used data from the larger, slower
memory below it. Understanding its structure — lines, sets, and ways — lets you predict hits and
misses and explains why certain access patterns are catastrophically slow.

## Cache lines: the unit of transfer

Memory doesn't move byte-by-byte between cache and DRAM. It moves in fixed **cache lines**,
typically **64 bytes** on x86-64. When you read a single `int`, the hardware fetches the entire
64-byte line containing it. This is why **spatial locality** pays off: the neighbors came along
for free.

## The cache hierarchy and AMAT

Real CPUs stack several caches, each bigger and slower than the last, before DRAM:

```
L1  ~32-64 KB   ~4 cycles   (per core, split I/D)
L2  ~256KB-1MB  ~12 cycles  (per core)
L3  ~8-64 MB    ~40 cycles  (shared across cores)
DRAM            ~200+ cycles (~60-100 ns)
```

The payoff of this staircase is captured by **Average Memory Access Time (AMAT)**:

```
AMAT = hit_time + miss_rate * miss_penalty
```

and it composes across levels (a miss at L1 "pays" L2's AMAT as its penalty):

```
AMAT = t_L1 + m_L1 * (t_L2 + m_L2 * (t_L3 + m_L3 * t_DRAM))
```

Because the DRAM penalty is ~50x an L1 hit, even a small miss-rate improvement at L1 can dominate
performance — the quantitative reason locality matters so much (next lesson).

## Sets, ways, and the address breakdown

A cache is organized as an array of **sets**, each containing some number of **ways** (line
slots). To find data, the hardware splits the address into three fields:

```
| tag | set index | block offset |
```

- **block offset** — which byte within the 64-byte line (low `log2(line_size)` bits; 6 bits for
  64-byte lines).
- **set index** — which set to look in (next `log2(num_sets)` bits).
- **tag** — the remaining high bits, stored alongside each cached line to confirm identity.

Lookup: use the set index to pick a set, then compare the tag against the (valid) lines in that
set. A match is a **hit**; otherwise a **miss** that fetches the line from below.

## Three geometries

- **Direct-mapped** (1 way per set): each address maps to exactly one slot. Simple and fast, but
  two hot addresses that map to the same set evict each other repeatedly.
- **Set-associative** (N ways): each set holds N lines; an address can live in any of the N
  slots, reducing conflicts. Most real caches are 4-, 8-, or 16-way.
- **Fully-associative** (one big set): any line anywhere. No conflict misses, but expensive to
  search; used only for tiny caches (like the TLB).

Within a set, when it's full, a **replacement policy** (often pseudo-LRU) chooses the victim.

## A worked decomposition

Consider a direct-mapped cache: 64-byte lines, 256 sets (so 16 KB total). Address breakdown:

- offset = 6 bits (64-byte line)
- index = 8 bits (256 sets = 2^8)
- tag = the rest

For address `0x1F40` = `0001 1111 0100 0000`:
- offset = low 6 bits = `000000` = 0
- index = next 8 bits = `01111101` = 125
- tag = remaining high bits.

So this access lands in set 125, byte 0. Two addresses with the **same index but different tag**
collide in a direct-mapped cache.

## Write policies (at the cache level)

Caches must decide what to do on writes — the same write-back/through choice you saw for the disk
buffer cache, one level up:

- **Write-back + write-allocate** (the common combo): a write updates the line in cache and marks it
  **dirty**; on a write miss, the line is first fetched, then written. Dirty lines flush to the next
  level on eviction. Absorbs repeated writes to the same line.
- **Write-through + no-write-allocate**: writes go straight to the next level; write misses don't pull
  the line in. Simpler, but more write traffic.

A **dirty bit** per line tracks whether a write-back is needed on eviction — a clean line can be
dropped for free.

## The three C's of misses

Every miss falls into one of three categories — a vocabulary for diagnosing performance:

- **Compulsory (cold)** — the first-ever access to a line; unavoidable without prefetching.
- **Capacity** — the working set is larger than the cache, so lines get evicted before reuse.
- **Conflict** — lines map to the same set and evict each other even though the cache isn't full
  (a direct-mapped/low-associativity artifact).

## Recap

- Data moves in 64-byte **lines**; reading one byte loads its whole line.
- Addresses split into tag / set index / block offset; lookup matches the tag within a set.
- Direct-mapped → set-associative → fully-associative trade simplicity for fewer conflicts.
- Misses are compulsory, capacity, or conflict — the diagnostic vocabulary for tuning.
