# HTTP Basics and Parsing

**HTTP** is the application protocol of the web, running over TCP. It's a simple, text-based
request/response protocol — which makes it a great parsing exercise and a frequent interview task.
Understanding its structure also reveals why framing (last lessons) matters so much.

## Request structure

An HTTP/1.1 request is text with `\r\n` line endings:

```
GET /index.html HTTP/1.1\r\n      <- request line: METHOD SP TARGET SP VERSION
Host: example.com\r\n             <- header: Name: Value
Accept: text/html\r\n             <- another header
\r\n                              <- blank line ends the headers
<optional body>
```

- The **request line** has three space-separated tokens: method, target (path), version.
- **Headers** follow, one `Name: Value` per line.
- A **blank line** (`\r\n` alone) terminates the header section; an optional body follows.

A response mirrors this: a **status line** (`HTTP/1.1 200 OK`), headers, blank line, body.

## Determining message length

Because TCP has no message boundaries, HTTP must tell the receiver where the body ends:

- **`Content-Length: N`** — exactly N bytes of body follow.
- **`Transfer-Encoding: chunked`** — the body arrives in size-prefixed chunks, ending with a
  zero-length chunk (used when the length isn't known up front, e.g., streaming).

Getting this wrong (e.g., trusting one and ignoring the other) leads to **request smuggling**
vulnerabilities — a real security concern.

## Methods, status codes, statelessness

- Common **methods**: GET (fetch), POST (submit), PUT (replace), DELETE, HEAD.
- **Status codes**: 2xx success, 3xx redirect, 4xx client error (404 not found), 5xx server error.
- HTTP is **stateless** — each request is independent; state is layered on via cookies/tokens. This
  statelessness is what lets servers scale horizontally (any server can handle any request).

## The evolution: HTTP/1.1 → 2 → 3

Each HTTP version attacked a performance bottleneck:

- **HTTP/1.0**: one request per TCP connection (open, request, response, close) — a new handshake
  every time.
- **HTTP/1.1**: **keep-alive** (reuse the connection for many requests) and chunked encoding. But
  responses come back **in order** on one connection — **head-of-line blocking** at the app layer.
- **HTTP/2**: **multiplexing** — many concurrent **streams** over one TCP connection, plus header
  compression (**HPACK**). Fixes app-layer HOL blocking, but a lost TCP segment still stalls all
  streams (TCP-layer HOL blocking).
- **HTTP/3**: runs over **QUIC** (on UDP), where streams are independent, so one lost packet only
  stalls its own stream — and connection setup + TLS are merged into fewer round trips.

**Caching** rides on headers too: `Cache-Control`, `ETag`, and conditional requests
(`If-None-Match` → `304 Not Modified`) avoid re-sending unchanged bodies.

## Methods: safe and idempotent

REST leans on two properties: a **safe** method has no side effects (GET, HEAD), and an **idempotent**
method gives the same result if repeated (GET, PUT, DELETE — but **not** POST). This matters for
retries: a client/proxy can safely retry an idempotent request after a network hiccup, but retrying a
POST might create a duplicate order.

## Parsing in practice

A parser reads the request line (split on spaces, validate the version), then reads headers
line-by-line until the blank line, splitting each on the first `:`. Robustness matters: reject
malformed request lines, handle missing headers, and never over-read the body. You'll implement a
request-line + header parser next.

## Recap

- HTTP requests: request line (method/target/version), headers (`Name: Value`), blank line, body.
- Body length comes from `Content-Length` or chunked encoding — not from TCP.
- Methods (GET/POST/...), status codes (2xx–5xx), and statelessness (sessions via cookies/tokens).
