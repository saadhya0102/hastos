# Limited Direct Execution

To virtualize the CPU among many processes, the OS faces a tension: it wants programs to run
**directly on the hardware** (for speed), but it must also stay **in control** (for protection and
fairness). The answer is **limited direct execution (LDE)** — run user code directly, but with
hardware-enforced limits and a way for the OS to regain the CPU.

## Direct execution, with limits

"Direct execution" means the CPU runs the program's actual instructions — no interpretation, full
speed. The "limited" part adds two safeguards:

- **Restricted operations** run only in kernel mode. A user program can't do privileged things
  (talk to devices, change page tables) directly — it must **trap** into the kernel via a system
  call (recall the ECF module).
- **The OS can take the CPU back.** Otherwise a process that never yields (an infinite loop) would
  hog the CPU forever.

## Regaining control: the timer interrupt

How does the OS get the CPU back from a running process? Not by asking nicely — by the **timer
interrupt**. The OS programs a hardware timer to fire every few milliseconds; when it does, control
traps into the kernel regardless of what the user program was doing. The kernel's handler can then
run the **scheduler** and possibly switch to another process.

This is **preemptive** scheduling: the OS forcibly preempts the running process at timer ticks. A
**cooperative** approach (relying on processes to yield) is fragile — one buggy loop hangs
everything.

## Putting it together

On a timer tick (or a syscall/blocking event): trap to kernel → save the current process's context
→ run the scheduler to pick the next process → restore its context → return to user mode. That
context switch (from the processes module) plus the timer is the complete mechanism. The *policy* —
*which* process to pick — is the scheduler, the subject of the rest of this module.

<Aside kind="insight">
Mechanism vs. policy: LDE + context switch + timer is the **mechanism** for sharing the CPU; the
scheduling algorithm (FIFO, RR, MLFQ...) is the **policy**. Separating the two is a recurring OS
design principle.
</Aside>

## Recap

- LDE runs user code directly but restricts privileged ops (trap to enter the kernel) and lets the
  OS reclaim the CPU.
- The timer interrupt enables preemption — the OS regains control even from non-yielding code.
- Mechanism (LDE + context switch + timer) is separate from policy (the scheduling algorithm).
