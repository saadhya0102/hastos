# Floating Point: IEEE-754

Real numbers are infinite and continuous; computer words are finite. **IEEE-754** is the
standard that squeezes an enormous range of real values into a fixed number of bits using a
binary form of scientific notation. Understanding it explains why `0.1 + 0.2 != 0.3`, why
`NaN != NaN`, and why floating-point comparisons need care.

## The layout of a 32-bit float

A `float` (single precision) splits its 32 bits into three fields:

```
 31  30        23 22                      0
 +--+-----------+-------------------------+
 |S |  exponent |        mantissa         |
 +--+-----------+-------------------------+
  1     8 bits          23 bits
```

- **S** — sign bit (0 = positive, 1 = negative).
- **exponent** — 8 bits, stored with a **bias** of 127.
- **mantissa (fraction)** — 23 bits, the fractional part of the significand.

The value (for normalized numbers) is:

```
value = (-1)^S  x  1.mantissa  x  2^(exponent - 127)
```

The leading `1.` is *implicit* — it isn't stored, which buys one extra bit of precision.

## The exponent bias

Storing the exponent biased (add 127) lets the same unsigned field represent both negative and
positive exponents while keeping bit-pattern ordering monotonic. A stored exponent of `127`
means an actual exponent of `0`; `128` means `+1`; `126` means `-1`.

So `1.0` is sign 0, stored exponent 127 (`0x7F`), mantissa 0:

```
0 01111111 00000000000000000000000  = 0x3F800000 = 1.0f
```

<MiniIDE language="c" starter={`#include <stdio.h>
#include <stdint.h>
#include <string.h>
int main(void) {
  float f = 1.0f;
  uint32_t bits;
  memcpy(&bits, &f, sizeof bits);   /* view the float's raw bits safely */
  printf("1.0f bits = 0x%08X\\n", bits);   /* 0x3F800000 */

  uint32_t s = bits >> 31;
  uint32_t e = (bits >> 23) & 0xFF;
  uint32_t m = bits & 0x7FFFFF;
  printf("sign=%u exp(stored)=%u exp(actual)=%d mant=0x%06X\\n", s, e, (int)e - 127, m);
  return 0;
}`} />

<Aside kind="pitfall">
To inspect a float's bits, copy them with `memcpy` (or a `union`) rather than
`*(uint32_t*)&f`. The pointer cast violates strict aliasing and is undefined behavior;
`memcpy` is the portable, well-defined way.
</Aside>

## Special values

Certain exponent patterns are reserved:

- **Zero**: exponent 0 and mantissa 0 (there's a +0 and a -0).
- **Subnormal (denormal)**: exponent 0, mantissa nonzero — no implicit leading 1; these
  represent tiny values near zero with reduced precision.
- **Infinity**: exponent all ones (255), mantissa 0. `+inf` and `-inf`.
- **NaN (Not a Number)**: exponent all ones, mantissa nonzero. Results from `0.0/0.0`,
  `sqrt(-1)`, etc.

## Why 0.1 + 0.2 != 0.3

Most decimal fractions have no exact binary representation, just as `1/3` has no exact decimal
representation. `0.1` becomes a repeating binary fraction that gets rounded to fit 23 mantissa
bits. The tiny rounding errors accumulate, so `0.1 + 0.2` lands just shy of `0.3`.

<MiniIDE language="c" starter={`#include <stdio.h>
int main(void) {
  double a = 0.1 + 0.2;
  printf("0.1 + 0.2 = %.17f\\n", a);     /* 0.30000000000000004 */
  printf("equal to 0.3? %d\\n", a == 0.3); /* 0 (false) */
  return 0;
}`} />

The lesson: **never compare floats with `==`.** Compare within a tolerance (epsilon):
`fabs(a - b) < 1e-9`. Choose epsilon based on the magnitudes involved.

## Precision and range

- `float` (32-bit): ~7 significant decimal digits.
- `double` (64-bit): ~15-16 significant decimal digits — the sensible default for most math.
- Beware catastrophic cancellation (subtracting nearly equal numbers loses precision) and
  adding numbers of very different magnitudes (the small one can vanish).

## Recap

- A float is sign + biased exponent + mantissa, value `(-1)^S x 1.mantissa x 2^(e-127)`.
- The leading 1 is implicit; the exponent is biased by 127.
- All-ones exponent encodes inf (mantissa 0) or NaN (mantissa nonzero); zero/subnormals use
  exponent 0.
- Many decimals aren't exact in binary, so never use `==`; compare within epsilon.
- Use `double` by default; inspect bits via `memcpy`, not aliasing casts.
