# Files, Directories, and Inodes

A file system turns a flat array of disk blocks into the familiar tree of named files. The central
data structure is the **inode**, and understanding what it holds (and what it doesn't) clears up a
surprising amount of Unix behavior — hard links, the difference between a name and a file, and how
paths are resolved.

## The inode

An **inode** ("index node") holds a file's **metadata** and the locations of its data blocks:

- size, owner, permissions, timestamps, link count, type;
- **block pointers** to where the file's data lives on disk.

Notice what's **not** in the inode: the **file's name**. A name is stored separately, in a
**directory**. The inode is the file; the name is just a label pointing to it.

## Directories are files that map names to inodes

A **directory** is itself a special file whose contents are a list of **(name → inode number)**
entries. Looking up `report.txt` in a directory means scanning its entries for that name to find
the inode number, then reading that inode.

This indirection is why a file can have **multiple names** (hard links) and why renaming or moving
within a filesystem is cheap (just edit directory entries, not the data).

## Path resolution

Resolving `/home/ann/report.txt` walks the tree, one component at a time:

1. Start at the root inode (a well-known inode number).
2. In root's directory, find `home` → its inode → read that directory.
3. Find `ann` → its inode → read that directory.
4. Find `report.txt` → its inode → now you can read the file.

Each step is a directory lookup followed by an inode read. (This is why deep paths and cold caches
cost multiple disk reads — and why the buffer cache, next lesson, matters.)

## Hard links vs. symbolic links

- A **hard link** is another directory entry pointing at the **same inode**. The inode's **link
  count** tracks how many names refer to it; the data is freed only when the count hits zero. All
  hard links are equal — there's no "original."
- A **symbolic (soft) link** is a tiny file whose contents are a **path string**. It points at a
  *name*, not an inode, so it can cross filesystems and can dangle if the target is removed.

## Block pointers: direct and indirect

A file's data blocks are referenced by pointers in the inode. To support both small and huge files
efficiently, classic inodes use a mix:

- **Direct pointers** — point straight at the first few data blocks (fast for small files).
- **Indirect pointers** — point at a block full of pointers (single indirect), or pointers to
  pointer-blocks (double/triple indirect) for very large files.

Small files need only direct pointers; large files chain through indirection — a tree that scales
to huge sizes while keeping small-file access cheap.

## Recap

- The inode holds metadata + data-block pointers — but **not** the name.
- Directories map names → inode numbers; this enables hard links and cheap renames.
- Path resolution walks component-by-component from the root, lookup then inode read.
- Hard links share an inode (link count); symlinks store a path. Direct + indirect pointers scale
  from tiny to huge files.
