# Files, Directories, and Inodes

A file system turns a flat array of disk blocks into the familiar tree of named files. The central
data structure is the **inode**, and understanding what it holds (and what it doesn't) clears up a
surprising amount of Unix behavior — hard links, the difference between a name and a file, how paths
are resolved, and why `fork` makes two processes share a file offset.

## On-disk layout

Before the inode, picture the whole disk (or partition) divided into fixed-size **blocks** (commonly
4 KB) grouped into regions. A classic Unix-style layout (à la the OSTEP "vsfs"):

```
| Super | inode  | data   | inode table          | data region                 |
| block | bitmap | bitmap | (array of inodes)    | (file + directory data)     |
```

- **Superblock**: filesystem-wide metadata — how many inodes/blocks, where each region starts, magic
  number. Read first at mount.
- **Inode bitmap / data bitmap**: one bit per inode / data block, marking free vs allocated.
  Allocating a file block = find a 0 bit, set it to 1.
- **Inode table**: a packed array of inodes. Inode number `N` lives at a computable offset, so
  "inode N" is a direct index — no search.
- **Data region**: the actual file contents and directory contents.

## The inode

An **inode** ("index node") holds a file's **metadata** and the locations of its data blocks:

- size, owner (uid/gid), permission bits, timestamps (atime/mtime/ctime), **link count**, type
  (regular file, directory, symlink, device...);
- **block pointers** to where the file's data lives on disk.

Notice what's **not** in the inode: the **file's name**. A name is stored separately, in a
**directory**. The inode is the file; the name is just a label pointing to it. Inodes are small
(commonly 128 or 256 bytes) so many pack into one block.

## Directories are files that map names to inodes

A **directory** is itself a special file whose contents are a list of **(name → inode number)**
entries. A common on-disk entry format records the inode number, entry length, name length, and the
name bytes:

```
inum | reclen | namelen | name
5    | 12     | 2       | "."
2    | 12     | 3       | ".."
13   | 16     | 8       | "report"
24   | 16     | 7       | "photos"
```

Looking up `report` means scanning entries for that name to get inode 13, then reading that inode.
Every directory has `.` (itself) and `..` (parent). This name/inode split is why a file can have
**multiple names** (hard links) and why renaming or moving *within a filesystem* is cheap — you edit
directory entries, not the data.

## Path resolution

Resolving `/home/ann/report.txt` walks the tree, one component at a time:

1. Start at the **root inode** (a well-known, fixed inode number, e.g. 2).
2. Read root's directory data, find `home` → its inode number → read that inode → read its data.
3. Find `ann` → its inode → read that directory.
4. Find `report.txt` → its inode → now you can read the file.

Each step is a directory lookup followed by an inode read. A deep path on a cold cache costs multiple
disk reads per component — which is exactly why the **buffer cache** (next lesson) and the **dentry
cache** exist.

## Hard links vs. symbolic links

- A **hard link** is another directory entry pointing at the **same inode**. The inode's **link
  count** tracks how many names refer to it; the data is freed only when the count hits **zero** (and
  no process still has it open). All hard links are equal — there's no "original." Hard links can't
  span filesystems (inode numbers are per-filesystem) and typically can't target directories.
- A **symbolic (soft) link** is a tiny file whose data is a **path string**. It points at a *name*,
  not an inode, so it can cross filesystems and can **dangle** if the target is removed.

This is why `rm` is really `unlink`: it removes a *directory entry* and decrements the link count.
The blocks come back only when the last link is gone.

## Block pointers: direct and indirect

A file's data blocks are referenced by pointers in the inode. To support both small and huge files
efficiently, classic inodes use a **multi-level index**:

- **Direct pointers** — point straight at the first few data blocks (fast for small files).
- **Single indirect** — points at a block full of pointers.
- **Double / triple indirect** — a pointer to a block of pointers to blocks of pointers, for very
  large files.

Small files need only direct pointers; large files chain through indirection — an unbalanced tree
that keeps small-file access cheap while scaling to enormous sizes. A quick capacity estimate with
4 KB blocks and 4-byte pointers (so **1024 pointers per block**):

```
direct (12 ptrs)         ->  12 * 4KB         =   48 KB
single indirect          ->  1024 * 4KB       =    4 MB
double indirect          ->  1024^2 * 4KB     =    4 GB
triple indirect          ->  1024^3 * 4KB     =    4 TB
```

Most files are small, so most accesses hit only direct pointers — the design optimizes the common
case.

## Open files: descriptors, the open-file table, and offsets

`open()` returns a small integer **file descriptor** (fd) — an index into the process's fd table.
There are three levels of indirection, and the middle one explains a lot:

```
per-process fd table   ->   system-wide open-file table   ->   inode (in-memory)
  fd 3  ------------------>  { offset, flags, refcount } ------> inode 13
```

- The **open-file entry** holds the current **read/write offset** and the open flags.
- `fork()` **shares** open-file entries (the child copies fd *numbers* pointing at the *same* entry),
  so parent and child **share one offset** — writes from both advance the same position (this is why
  shell redirection interleaves correctly).
- `dup()`/`dup2()` also share an entry; two separate `open()`s of the same file get **separate**
  entries (independent offsets) but the same inode.

## Recap

- The disk is laid out as superblock + bitmaps + inode table + data region.
- The inode holds metadata + multi-level (direct/indirect) data-block pointers — but **not** the
  name.
- Directories map names → inode numbers; this enables hard links and cheap renames.
- Path resolution walks component-by-component from the root, lookup then inode read.
- Hard links share an inode (link count → 0 frees data); symlinks store a path.
- fd → open-file entry (offset/flags) → inode; `fork`/`dup` share the entry (shared offset).
