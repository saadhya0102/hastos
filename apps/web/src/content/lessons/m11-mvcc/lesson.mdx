# MVCC and Isolation

Databases must let many transactions run **concurrently** while each sees a **consistent** view of
the data. **Multi-Version Concurrency Control (MVCC)** is the dominant technique: instead of locking
readers out, the database keeps **multiple versions** of each row so readers and writers rarely
block each other. This connects the concurrency module to real database design.

## The problem: readers vs. writers

With simple locking, a long-running read query would block writers (or vice versa), killing
concurrency. We want: **readers never block writers, and writers never block readers**, while each
transaction still sees a coherent snapshot — not a half-applied mix of others' changes.

## The idea: keep multiple versions

When a transaction updates a row, MVCC doesn't overwrite it in place — it writes a **new version**
tagged with the transaction's timestamp/id, leaving the old version intact. Each row effectively has
a chain of versions over time.

A transaction reads as of a **snapshot** (a point in time): for each row, it sees the **latest
version that was committed before its snapshot** — ignoring versions created by transactions that
hadn't committed yet. So a reader gets a stable, consistent view even as writers create newer
versions concurrently.

## Snapshot isolation

The isolation level MVCC naturally provides is **snapshot isolation**: each transaction operates
against a frozen snapshot from its start. Reads are repeatable and never see partial updates. It's
strong and cheap, though not fully serializable — the classic gap is **write skew**, where two
transactions read overlapping data and make disjoint writes that together violate an invariant
(serializable MVCC adds extra checks to catch this).

## Garbage collection of old versions

Keeping every version forever wastes space, so the database **reclaims** versions no longer visible
to any active transaction (e.g., PostgreSQL's `VACUUM`). This is **space amplification** in action —
the cost MVCC pays for non-blocking concurrency, analogous to LSM compaction reclaiming overwritten
keys.

## Recap

- MVCC keeps multiple timestamped versions so readers and writers don't block each other.
- A transaction reads the latest version committed before its snapshot → snapshot isolation.
- Snapshot isolation prevents dirty/non-repeatable reads but allows write skew (serializable adds
  checks).
- Old versions are garbage-collected (e.g., VACUUM) to bound space amplification.
