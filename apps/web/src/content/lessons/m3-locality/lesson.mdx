# Locality and Cache-Friendly Code

Caches reward predictable, reuse-heavy access. The two properties that make code fast are
**temporal locality** (reuse the same data soon) and **spatial locality** (use nearby data
soon). Most practical optimization is about increasing one or both.

## Temporal and spatial locality

- **Temporal locality**: if you touch a value, you'll likely touch it again soon — so keep it in
  cache by reusing it before it's evicted (e.g., hoist a repeatedly-read value into a local).
- **Spatial locality**: if you touch an address, you'll likely touch its neighbors soon — so lay
  data out contiguously and walk it in order, using the whole 64-byte line you paid to load.

## Row-major vs. column-major: the canonical example

C stores 2D arrays in **row-major** order: `a[i][j]` is at `i*ncols + j`, so an entire row is
contiguous. Traversal order therefore matters enormously:

```c
// FAST: inner loop walks contiguous memory (good spatial locality)
for (int i = 0; i < n; i++)
  for (int j = 0; j < n; j++)
    sum += a[i][j];        // a[i][0], a[i][1], ... are adjacent

// SLOW: inner loop jumps by a full row each step (stride n)
for (int j = 0; j < n; j++)
  for (int i = 0; i < n; i++)
    sum += a[i][j];        // a[0][j], a[1][j], ... are n*4 bytes apart
```

The two loops do the *same* arithmetic and the *same* number of accesses. The first can be many
times faster because it uses each cache line fully; the second touches one element per line and
thrashes.

## Stride and cache-line utilization

**Stride** is the byte gap between consecutive accesses. Stride-1 (contiguous) uses all 16 ints
of a 64-byte line before moving on — 1 miss per 16 elements. A large stride (e.g., one column of
a big matrix) touches a single element per line — 1 miss per element, 16x more memory traffic.

<Aside kind="insight">
Rule of thumb: aim for small, regular strides. Every cache line you load costs ~100 ns; using
only 4 of its 64 bytes wastes 94% of that cost. "Cache-friendly" largely means "use the whole
line you already paid for."
</Aside>

## The hardware prefetcher

Modern CPUs watch your access pattern and **prefetch** ahead: if you're marching through memory at a
regular stride, the prefetcher fetches the *next* lines before you ask, hiding much of the DRAM
latency. This is a second reason to prefer small, regular strides — a **predictable** pattern lets the
prefetcher work; random pointer-chasing defeats it (nothing to predict). You can also issue explicit
software hints (`__builtin_prefetch`) for irregular-but-known-ahead accesses, though it's a
last-resort tool.

## Data layout: AoS vs. SoA

How you group fields affects locality:

- **Array of Structs (AoS)**: `struct P { float x, y, z; } pts[N];` — good when you use all fields
  of each element together.
- **Struct of Arrays (SoA)**: `float xs[N], ys[N], zs[N];` — good when you sweep one field across
  all elements (e.g., sum all `x`), because that field is contiguous.

If a hot loop reads only `x` from a million points, SoA keeps the loaded lines full of `x`
values; AoS wastes two-thirds of each line on `y`/`z` you didn't need.

## Practical techniques

- Traverse in memory order (match loop nesting to layout).
- Hoist repeatedly-used values into locals/registers (temporal locality).
- Shrink hot structures so more fit per line and in cache.
- Choose AoS vs SoA based on the dominant access pattern.

## Recap

- Temporal locality = reuse soon; spatial locality = use neighbors soon.
- Row-major C arrays: keep the contiguous index innermost.
- Small strides use whole cache lines; large strides waste most of each line.
- AoS vs SoA: lay data out to match how the hot loop reads it.
