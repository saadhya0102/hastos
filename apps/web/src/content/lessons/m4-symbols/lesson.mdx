# Symbols: Strong, Weak, and Resolution

The linker's central job is **symbol resolution**: matching every *use* of a name to exactly one
*definition*. To do that it reads each object file's **symbol table** and applies a small set of
rules. Knowing those rules explains a class of subtle, dangerous bugs — especially when
duplicate definitions silently pick the wrong one.

## What is a symbol?

When the assembler produces an object file (`.o`), it records a **symbol table**: a list of names
the file is involved with. Each symbol is one of:

- **Defined (global)** — this file provides it (a non-static function or global variable).
- **Undefined (referenced)** — this file *uses* it but doesn't define it; the linker must find a
  definition elsewhere (e.g., `printf`).
- **Local** — `static` functions/variables, visible only within this file; the linker doesn't
  resolve these across files.

You can inspect them with `nm file.o`: `T` = defined in text, `U` = undefined, `t` = local.

## Strong vs. weak symbols

Global symbols come in two strengths:

- **Strong** — functions and **initialized** global variables (`int g = 5;`).
- **Weak** — **uninitialized** global variables (`int g;` at file scope, a tentative definition),
  and anything explicitly marked weak (`__attribute__((weak))`).

This strength drives how the linker handles duplicates.

## The resolution rules

When the linker sees the same global name defined in more than one place, it applies three rules:

1. **Multiple strong definitions → error.** Two `int g = 0;` (or two function bodies with the
   same name) across files is a `multiple definition` link error.
2. **One strong + any number of weak → the strong wins.** No error; all references bind to the
   strong definition.
3. **Multiple weak, no strong → the linker picks one** (arbitrarily, often the first/largest).
   No error — but this is a silent trap.

And of course:
4. **A referenced symbol with no definition anywhere → `undefined reference` error.**

## Why this is dangerous

Rules 2 and 3 mean the linker can **silently** bind a name to a definition you didn't intend.
Classic disaster:

```c
// a.c
int x;            // weak (uninitialized) — programmer thinks it's an int
// b.c
double x = 3.14;  // strong — a DIFFERENT type, same name
```

The linker happily binds `x` in `a.c` to the 8-byte `double` in `b.c` (rule 2: strong wins).
Now `a.c` reads/writes 4 bytes thinking it's an `int`, corrupting the `double` — a bug with no
error message, just mysterious memory corruption. This is exactly why you keep globals `static`
or declare them `extern` in a shared header and define them once.

<Aside kind="pitfall">
Treat duplicate global names as a hazard, not a convenience. Enable `-fno-common` (now the
default in modern GCC/Clang) so tentative definitions become errors, and keep file-private state
`static`.
</Aside>

## Recap

- Object files carry a symbol table: defined, undefined, or local symbols.
- Strong = functions + initialized globals; weak = uninitialized globals.
- Resolution: two strong = error; strong+weak = strong wins; multiple weak = pick one;
  no definition = undefined reference.
- Strong-beats-weak can silently alias mismatched types — keep globals `static`/`extern` and
  define once.
