# False Sharing

Caching usually helps, but in multithreaded code it can quietly destroy performance through
**false sharing** — when independent variables used by different threads happen to live on the
same cache line, forcing the cores to fight over it.

## Cache coherence, briefly

In a multicore CPU, each core has its own L1/L2 cache, and hardware keeps them **coherent**: if
one core writes a cache line, other cores' copies of that line are invalidated so nobody reads
stale data. Coherence operates at **cache-line granularity** (64 bytes) — not per variable.

That last point is the trap.

## How coherence works: MESI

Coherence is usually implemented with a protocol like **MESI**, giving each cached line one of four
states:

- **Modified** — this core has the only copy and it's dirty (differs from memory).
- **Exclusive** — this core has the only copy and it's clean.
- **Shared** — multiple cores may hold clean read-only copies.
- **Invalid** — the copy is stale/unusable.

A **write** requires the line in Modified state, which means invalidating every other core's copy
first (a "read-for-ownership"). So two cores repeatedly writing the *same line* force it to bounce:
core 1 takes it Modified (invalidating core 2), core 2 then takes it Modified (invalidating core 1),
over and over — the **ping-pong**. Reads shared across cores are fine (all can hold it Shared); it's
concurrent **writes** to the same line that hurt.

## The false-sharing scenario

```c
struct { long a; long b; } counters;   // a and b are 8 bytes apart -> same 64-byte line

// Thread 1 hammers counters.a; Thread 2 hammers counters.b
```

`a` and `b` are *logically independent* — the threads never touch each other's variable. But
they sit on the **same cache line**. Every time thread 1 writes `a`, it invalidates thread 2's
cached copy of the line (which holds `b`), and vice versa. The line "ping-pongs" between cores,
turning what should be fast local writes into a storm of coherence traffic. Throughput can drop
by 5–10x with *no* logical contention.

## The fix: pad to separate lines

Ensure each hotly-written variable lives on its own cache line, using alignment/padding:

```c
#include <stdalign.h>

struct PaddedCounter {
  alignas(64) long value;   // force each counter onto its own 64-byte line
};
struct PaddedCounter counters[2];   // counters[0] and counters[1] never share a line
```

Or pad inside a struct:

```c
struct { long a; char pad[56]; long b; } counters;  // a and b now 64 bytes apart
```

After padding, each thread owns its line exclusively — no invalidation traffic, full speed.

<Aside kind="warning">
Padding trades memory for speed and should target only **hot, concurrently-written** data
(per-thread counters, queue head/tail pointers). Don't blanket-pad everything — it wastes cache
capacity and can *hurt* by reducing how much fits in cache.
</Aside>

## True vs. false sharing, and detecting it

- **False sharing**: threads touch *different* variables that share a line — fixable by padding.
- **True sharing**: threads genuinely contend on the *same* variable — padding won't help; you need a
  better algorithm (per-thread accumulation then combine, or a lock-free/sharded design).

To find it in practice, profilers surface cache-line contention: Linux `perf c2c` (cache-to-cache)
reports **HITM** events (a load that hit a line Modified in *another* core's cache) and points at the
offending line and offsets. A sudden speed collapse when you add threads — with no shared logical
state — is the classic false-sharing fingerprint.

## Where it shows up

- Per-thread statistics/counters packed in an array.
- A lock-free queue's `head` and `tail` on the same line (producers and consumers fight).
- Flags or state for different threads grouped in one struct.

This connects directly to the concurrency module: when you build a lock-free queue later, you'll
pad `head` and `tail` apart precisely to avoid this.

## Recap

- Coherence tracks ownership per **cache line**, not per variable.
- False sharing: independent variables on one line cause cores to invalidate each other's caches,
  tanking throughput with no logical contention.
- Fix by aligning/padding hot, concurrently-written data onto separate lines — selectively.
