# TCP vs. UDP

The two transport protocols sit on top of IP and offer opposite trade-offs. **TCP** is a reliable,
ordered, connection-oriented byte stream; **UDP** is a thin, best-effort, message-oriented layer.
Choosing correctly — and knowing TCP's mechanisms — is essential systems and interview knowledge.

## TCP: reliable, ordered, connection-oriented

TCP guarantees that bytes arrive **in order**, **without loss or duplication**, on an established
**connection**. It achieves this with sequence numbers, acknowledgments, retransmission of lost
segments, and checksums. You get a clean byte stream — at the cost of connection setup, per-segment
acknowledgment overhead, and head-of-line blocking.

## UDP: best-effort datagrams

UDP just sends **datagrams** (messages) with almost no machinery: no connection, no ordering, no
retransmission, no congestion control. A datagram may be lost, duplicated, or arrive out of order.
In exchange you get low overhead and latency, and message boundaries are preserved (one `sendto` =
one `recvfrom`). Apps that need reliability build it themselves on top (e.g., QUIC, game netcode).

## The TCP handshake and teardown

A TCP connection opens with a **three-way handshake**:

```
client -> SYN ->         server
client <- SYN-ACK <-     server
client -> ACK ->         server   (connection established)
```

It closes with a four-way FIN/ACK exchange (each side closes its half), often leaving the initiator
in **TIME_WAIT** briefly to ensure stray packets drain.

## Flow control, congestion control, and Nagle

- **Flow control** (receiver-driven): the receiver advertises a **window** of how much it can
  buffer, so a fast sender won't overwhelm a slow receiver.
- **Congestion control** (network-driven): TCP probes available bandwidth (slow start, congestion
  avoidance) and backs off on loss, sharing the network fairly.
- **Nagle's algorithm**: coalesces small writes into fewer packets to reduce overhead — but can add
  latency for interactive/small-message workloads. Disable it with `TCP_NODELAY` when latency
  matters.

## TCP is a byte stream: you must frame messages

A subtle but critical point: TCP has **no message boundaries**. If the sender does two `write`s of
100 bytes, the receiver might `read` 200 at once, or 37 then 163 — TCP only promises the *bytes* arrive
in order. So application protocols must add their own **framing**:

- **Length prefix**: send a fixed-size length, then that many bytes (simple, robust).
- **Delimiter**: end each message with a marker like `\r\n` (HTTP headers, Redis).

Forgetting this yields the classic bug where a parser "usually works" (small messages arrive in one
read) then breaks under load or large payloads. UDP, by contrast, *preserves* boundaries (one
`sendto` = one `recvfrom`), but caps message size and can drop them.

## Congestion control in a bit more depth

TCP's congestion control keeps the internet from collapsing by adapting a **congestion window
(cwnd)**:

- **Slow start**: begin small, **double** cwnd each round-trip (exponential) to quickly find
  capacity.
- **Congestion avoidance**: past a threshold, grow **linearly** (additive increase).
- **On loss**: cut cwnd sharply (multiplicative decrease) — the **AIMD** pattern (additive-increase,
  multiplicative-decrease) that converges to a fair share.
- **Fast retransmit/recovery**: three duplicate ACKs signal a lost segment; resend it immediately
  without waiting for a timeout.

Loss is treated as the congestion signal, which is why a single lossy link throttles throughput, and
why **head-of-line blocking** (one lost segment stalls everything behind it in the stream) motivated
**QUIC** (HTTP/3) to run independent streams over UDP.

## Choosing a transport

- **TCP**: when you need reliability and ordering — web, APIs, file transfer, databases.
- **UDP**: when low latency beats reliability, or you'll build your own reliability — DNS, video/
  voice, gaming, QUIC/HTTP-3.

## Recap

- TCP: reliable, ordered, connection-oriented byte stream (handshake, ACKs, retransmit) — higher
  overhead.
- UDP: best-effort datagrams, message boundaries, minimal overhead — build reliability yourself if
  needed.
- TCP adds flow control (window), congestion control (slow start/backoff), and Nagle (disable with
  TCP_NODELAY for latency).
- Pick TCP for correctness-critical streams, UDP for latency-critical/loss-tolerant traffic.
