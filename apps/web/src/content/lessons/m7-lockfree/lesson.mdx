# Lock-Free Programming and the ABA Problem

The capstone of concurrency: building data structures that are correct **without locks**, using
only atomic operations. Lock-free code can offer better scalability and avoids deadlock entirely,
but it's among the hardest code to get right — every interleaving and memory-ordering subtlety
matters. This is the flagship interview territory.

## What "lock-free" means

A data structure is **lock-free** if **at least one** thread always makes progress in a bounded
number of steps, regardless of how others are scheduled (so no deadlock, and a stalled thread can't
block everyone). Stronger is **wait-free** (every thread makes progress in bounded steps). Weaker
is **obstruction-free**. The everyday goal is lock-free via **CAS loops**.

## The CAS loop

The fundamental lock-free idiom: read the current value, compute the new value, and **CAS** it in —
retrying if someone changed it underneath you.

```c
// atomic increment via CAS (illustrative; fetch_add is better in practice)
int old = atomic_load(&x);
while (!atomic_compare_exchange_weak(&x, &old, old + 1)) {
  /* old was updated to the current value; loop retries with fresh old */
}
```

If another thread modified `x` between the load and the CAS, the CAS fails, `old` is refreshed, and
you retry. Progress is guaranteed because a failure means *someone else* succeeded.

## The Treiber stack and where it breaks

A classic lock-free stack pushes by CAS-ing the head:

```c
void push(Node *n) {
  do { n->next = atomic_load(&head); }
  while (!atomic_compare_exchange_weak(&head, &n->next, n));
}
```

Pop is symmetric: read `head`, CAS `head` to `head->next`. It looks correct — but pop hides a
vicious bug: the **ABA problem**.

## The ABA problem

CAS only checks that a value **equals** the expected one — not that it never changed. Consider pop:

1. Thread 1 reads `head = A`, plans to set `head = A->next = B`. It pauses.
2. Thread 2 pops A, pops B, then pushes A back. Now `head = A` again — but `A->next` is no longer B!
3. Thread 1 resumes; its CAS sees `head == A` (expected!) and **succeeds**, setting `head = B` — a
   node that was already removed/freed. Corruption or use-after-free.

The value went **A → B → A**, so CAS was fooled into thinking nothing changed. This is the **ABA
problem**, the signature hazard of naive lock-free code.

## Mitigating ABA

- **Tagged pointers / version counters** — pack a monotonically increasing tag with the pointer and
  CAS both together (a double-width CAS). The tag changes on every update, so A-with-tag-1 ≠
  A-with-tag-3.
- **Hazard pointers** — threads publish the pointers they're using; memory isn't reclaimed while
  referenced, preventing the "freed then reused" reappearance.
- **Epoch-based reclamation (RCU-style)** — defer freeing until no thread could still hold a stale
  reference.
- **Avoid the hazard** — restrict the problem (e.g., single consumer) so reclamation races can't
  occur.

## The MPSC simplification

A general multi-producer/multi-consumer lock-free queue is very hard (reclamation, ABA). But many
real systems need only **multi-producer, single-consumer (MPSC)**: many threads enqueue, exactly
**one** thread dequeues. The single-consumer restriction eliminates consumer-side races and much of
the ABA hazard, making a correct lock-free queue tractable — the producers CAS to publish nodes,
and the lone consumer walks the list without competing dequeuers.

## Progress guarantees: a precise vocabulary

"Non-blocking" comes in strengths, worth stating precisely:

- **Wait-free** (strongest): *every* thread finishes its operation in a bounded number of steps —
  no starvation, ever. Hardest to build.
- **Lock-free**: the *system* makes progress (some thread always completes), but an individual thread
  can be starved by others repeatedly winning the CAS race. A CAS-retry loop is lock-free.
- **Obstruction-free** (weakest): a thread finishes if it eventually runs alone (no contention).
- **Blocking**: a stalled thread (holding a lock) can halt everyone — what locks are.

## Safe memory reclamation

The deep reason lock-free `free` is hard: after you unlink a node, **another thread may still be
reading it**, so you can't immediately free it (use-after-free) — but you also can't leak it. The
three standard solutions:

- **Hazard pointers**: each thread publishes the pointers it's currently using; a node is freed only
  once no thread's hazard pointers reference it.
- **Epoch-based reclamation (EBR)**: threads announce an epoch on entry; memory retired in epoch N is
  freed only after all threads have advanced past N.
- **RCU (Read-Copy-Update)**: readers run lock-free in read-side critical sections; writers publish new
  versions and defer reclamation until a **grace period** (after which no reader can hold the old
  version) — the backbone of scalable read-mostly structures in the Linux kernel.

## When to go lock-free

Rarely. Locks are simpler, easier to verify, and fast enough for the vast majority of code.
Lock-free is justified for hot, highly-contended structures where lock overhead or
priority-inversion/deadlock risk genuinely hurts — and only with rigorous testing (TSan, stress
tests, many seeds), because bugs are nondeterministic and catastrophic.

## Recap

- Lock-free = guaranteed system-wide progress without locks, via CAS loops; immune to deadlock.
- The Treiber stack shows the **ABA problem**: CAS checks equality, not history (A→B→A fools it).
- Mitigate with tagged pointers/version counters, hazard pointers, epochs, or by restricting the
  problem (e.g., single consumer).
- MPSC is the tractable sweet spot; prefer locks unless measurement demands lock-free.
