# From Source to a Running Program: The Toolchain

When you click **Submit** on HastOS, your C source doesn't run directly — it's transformed
through a pipeline of tools into machine code the CPU can execute. Understanding this pipeline
demystifies a huge class of errors ("why won't this compile?" vs. "why won't this link?") and
sets up the Linking module later.

## The four stages

A single command like `gcc main.c -o app` actually runs four stages in sequence:

```
main.c
  | 1. Preprocess  (cpp)        -> expands #include, #define, #if
  v
main.i  (pure C, no directives)
  | 2. Compile     (cc1)        -> translates C to assembly
  v
main.s  (x86-64 assembly)
  | 3. Assemble    (as)         -> assembly to machine code
  v
main.o  (object file: machine code + a symbol table)
  | 4. Link        (ld)         -> combine objects + libraries
  v
app  (executable)
```

### 1. Preprocess
The preprocessor is a text transformer. It pastes in the contents of `#include` headers,
substitutes `#define` macros, and resolves conditional `#if`/`#ifdef` blocks. It doesn't
understand C — it just manipulates text. The output is a single, larger `.i` file of pure C.

### 2. Compile
The compiler proper parses that C, type-checks it, optimizes, and emits **assembly** for your
target architecture (x86-64 here). This is where *most* of your errors are caught: syntax
errors, type mismatches, undeclared variables.

### 3. Assemble
The assembler turns human-readable assembly into binary **machine code**, producing an
**object file** (`.o`). An object file contains machine code plus a **symbol table**: a list
of the names it *defines* (e.g., `my_strlen`) and the names it *needs but doesn't have yet*
(e.g., `printf`).

### 4. Link
The linker stitches multiple object files and libraries together, **resolving symbols**:
matching each "I need `printf`" to a definition (in the C standard library). The result is a
runnable executable. Unresolved symbols cause *link-time* errors.

<Aside kind="insight">
Key distinction: the **compiler** works on one translation unit at a time and never sees other
files. The **linker** is what connects them. That's why calling a function you declared but
never defined compiles fine but fails to link.
</Aside>

## Compile-time vs. link-time errors

Telling these apart saves hours:

- **Compile error** — the compiler rejected your C. Examples: `expected ';'`, `unknown type
  name 'itn'`, `incompatible pointer types`. Fix the *code in that file*.
- **Link error** — compilation succeeded for each file, but the linker couldn't resolve a
  symbol. The classic messages:
  - `undefined reference to 'foo'` — you declared/called `foo` but no object defines it.
  - `multiple definition of 'bar'` — two files both define `bar`.

```c
// Compiles fine (declaration is enough for the compiler)...
int compute(int);
int main(void) { return compute(3); }
// ...but FAILS TO LINK: "undefined reference to 'compute'"
// because no file actually defines compute().
```

## Declarations vs. definitions

This is the heart of why C splits into `.h` and `.c` files:

- A **declaration** says "this name exists and has this type" — e.g., a function prototype
  `size_t my_strlen(const char *);`. It's a promise. Declarations go in **headers**.
- A **definition** provides the actual body/storage — `size_t my_strlen(const char *s) { ... }`.
  Definitions go in **`.c` files**.

You can declare something many times but must define it exactly once (the "one definition
rule"). Headers let many files share declarations without duplicating definitions.

## Warnings and optimization flags

Two flag groups you'll use constantly:

- **Warnings**: `-Wall -Wextra` enable most useful warnings. Treat warnings as bugs —
  uninitialized variables and suspicious conversions usually surface here first.
- **Optimization**: `-O0` (none, best for debugging), `-O1`/`-O2` (typical release), `-O3`
  (aggressive). Higher levels can *expose* undefined behavior that "worked" at `-O0`.

<Aside kind="note">
When the grader supports it (a Judge0 backend), HastOS compiles submissions with sanitizers like
`-fsanitize=address,undefined`, which instrument your program to catch out-of-bounds accesses,
use-after-free, and UB at runtime — turning silent corruption into a clear report. The lightweight
Piston grader can't inject those flags, so it still checks correctness via the test output; either
way, writing UB-free code is on you.
</Aside>

## Inspecting the pipeline yourself

You can stop the pipeline at any stage and look — a superpower for debugging and learning:

```
gcc -E main.c              # stop after preprocessing (see expanded macros/includes)
gcc -S main.c              # stop after compile: emit main.s (read the assembly!)
gcc -c main.c              # stop after assemble: emit main.o (object file)
nm main.o                  # list symbols: T=defined text, U=undefined, t=local
objdump -d main.o          # disassemble machine code back to assembly
readelf -h a.out           # inspect the ELF header/segments
```

Two runtime debuggers you'll reach for constantly:

- **gdb** — set breakpoints, step, inspect variables and the stack (`bt` for a backtrace on a crash).
- **valgrind** / **AddressSanitizer** — catch memory errors (leaks, out-of-bounds, use-after-free).

## Undefined vs. implementation-defined vs. unspecified

Not all "it depends" is the same, and the distinction matters:

- **Undefined behavior (UB)**: anything may happen (out-of-bounds, signed overflow). Never rely on it.
- **Implementation-defined**: the implementation must pick and **document** a behavior (e.g.,
  `sizeof(int)`, whether `char` is signed). Portable code shouldn't assume a specific choice.
- **Unspecified**: the implementation picks but need not document or be consistent (e.g., the
  evaluation order of function arguments).

## How HastOS compiles your work

For a `harness`-graded problem, your editable file (say `strlen.c`) is compiled together with
a hidden **driver** that calls your function and checks results, plus the shared **header**.
Roughly:

```
# On a sanitizer-capable grader (Judge0):
gcc -std=c11 -O1 -g -fsanitize=address,undefined  main.c  -o app
# On Piston: same, minus the injected flags. main.c is the header + your code + driver
# assembled into one translation unit.
./app   # prints machine-readable PASS/FAIL lines the grader parses
```

If your code doesn't compile, you'll see the compiler's own error output. If it compiles but
a test fails, you'll see which test and why.

## Recap

- Source becomes an executable via preprocess → compile → assemble → link.
- The compiler sees one file; the linker connects them and resolves symbols.
- Compile errors are about code in a file; link errors are about missing/duplicate definitions.
- Declarations (headers) promise; definitions (.c files) deliver — define exactly once.
- Use `-Wall -Wextra`, understand `-O` levels, and trust sanitizers.
