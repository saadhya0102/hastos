# Signals and Async-Safety

A **signal** is a small asynchronous notification the kernel delivers to a process — a software
analog of a hardware interrupt. They power Ctrl-C, child-death notifications, timers, and crash
reporting. They're also a notorious source of subtle bugs because a handler can fire *between any
two instructions* of your program.

## Signals as software interrupts

Each signal has a number and a default action. Common ones:

- `SIGINT` (2) — interrupt from the keyboard (Ctrl-C); default: terminate.
- `SIGTERM` (15) — polite termination request.
- `SIGKILL` (9) — forced kill; **cannot** be caught or ignored.
- `SIGSEGV` (11) — invalid memory access; default: terminate (+ core dump).
- `SIGCHLD` (17) — a child stopped or terminated (used to reap children).
- `SIGALRM` (14) — a timer expired.

A process can **catch** most signals by installing a **handler**, **ignore** them, or accept the
**default**.

## Installing a handler

Use `sigaction` (preferred over the older `signal`):

```c
#include <signal.h>
volatile sig_atomic_t got_sigint = 0;

void on_sigint(int sig) { got_sigint = 1; }   // keep handlers tiny!

int main(void) {
  struct sigaction sa = {0};
  sa.sa_handler = on_sigint;
  sigaction(SIGINT, &sa, NULL);
  while (!got_sigint) { /* work */ }
  return 0;
}
```

When `SIGINT` arrives, normal flow is suspended, the handler runs, then flow resumes. The handler
can fire at essentially any point — which is the source of danger.

## Async-signal-safety

Because a handler can interrupt the main program mid-operation, it must only call
**async-signal-safe** functions — ones safe to invoke from a handler. Critically, most of the
standard library is **not** safe: `printf`, `malloc`, and friends use shared internal state and
locks; if a signal hits while `malloc` holds its lock and the handler also calls `malloc`, you can
**deadlock or corrupt the heap**.

Safe functions are a short list (e.g., `write`, `_exit`, `signal-safe` syscalls). The disciplined
pattern: **handlers do almost nothing** — set a `volatile sig_atomic_t` flag (or `write` a byte to
a self-pipe) and let the main loop do the real work.

## sig_atomic_t and volatile

A flag shared between a handler and main code must be `volatile sig_atomic_t`:

- **`sig_atomic_t`** — a type guaranteed to be read/written atomically (no torn access) with
  respect to signals.
- **`volatile`** — tells the compiler the value can change asynchronously, so it won't cache it in
  a register or optimize the check away.

```c
volatile sig_atomic_t stop = 0;
void handler(int s){ stop = 1; }
/* main loop checks `stop` */
```

## EINTR: interrupted system calls

A signal can interrupt a blocking syscall (like `read`), which then returns -1 with `errno ==
EINTR` — *before* doing its work. Robust code retries:

```c
ssize_t n;
do { n = read(fd, buf, len); } while (n < 0 && errno == EINTR);
```

Or install handlers with `SA_RESTART` so the kernel auto-restarts many interrupted calls.

## Blocking signals and the self-pipe trick

You don't always want a signal to fire *right now*. Each process/thread has a **signal mask** —
signals that are temporarily **blocked** stay **pending** until unblocked (managed with
`sigprocmask`/`pthread_sigmask`). This lets you protect a critical section from a handler running in
the middle of it.

But the deeper problem is mixing signals with an **event loop** (`select`/`poll`/`epoll`): a signal
arriving while you're blocked in `select` is awkward, and handlers can't safely do real work. The
classic fix is the **self-pipe trick**: the handler just does one async-signal-safe `write` of a byte
to a pipe; the event loop watches the pipe's read end like any other fd, so the signal becomes a
normal, synchronous "readable fd" event handled in your main loop. (Linux's `signalfd` packages this
idea directly.)

## Recap

- Signals are asynchronous software interrupts; some (SIGKILL/SIGSTOP) can't be caught.
- Handlers can fire between any instructions, so they must be tiny and async-signal-safe — avoid
  `printf`/`malloc`; prefer setting a flag or `write` to a self-pipe.
- Share state via `volatile sig_atomic_t`.
- Handle `EINTR` by retrying interrupted syscalls (or use `SA_RESTART`).
